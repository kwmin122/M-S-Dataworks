# Software Design Document (SDD) 작성 스킬

코드 작성 전 설계 문서를 먼저 작성할 때 활성화됩니다.

## SDD 템플릿

### 1. 개요 (Overview)
- **문제 정의**: 무엇을 해결하는가
- **목표**: 달성하려는 것
- **비목표 (Non-goals)**: 이 설계에서 다루지 않는 것
- **배경**: 관련 컨텍스트, 이전 결정사항

### 2. 아키텍처 (Architecture)
- **시스템 구조**: 컴포넌트 간 관계
- **데이터 흐름**: 요청이 어떻게 처리되는지
- **기술 선택 근거**: 왜 이 기술을 선택했는지

### 3. 데이터 모델 (Data Model)
- **엔티티 정의**: 주요 데이터 구조
- **관계**: 엔티티 간 관계 (1:N, N:M 등)
- **인덱스 전략**: 쿼리 최적화를 위한 인덱스

### 4. API 설계 (API Design)
- **엔드포인트 목록**: HTTP 메서드, 경로, 설명
- **요청/응답 스키마**: 각 엔드포인트의 입출력
- **에러 응답**: 에러 코드와 메시지 형식
- **인증/인가**: 접근 제어 방식

### 5. 보안 고려사항 (Security)
- **인증 방식**: JWT, OAuth, 세션 등
- **인가 모델**: RBAC, ABAC 등
- **데이터 보호**: 암호화, 마스킹
- **입력 검증**: Validation 규칙

### 6. 테스트 전략 (Testing Strategy)
- **단위 테스트**: 핵심 비즈니스 로직
- **통합 테스트**: API, DB 연동
- **E2E 테스트**: 주요 시나리오
- **성능 테스트**: 부하 테스트 기준

### 7. 배포 전략 (Deployment)
- **CI/CD 파이프라인**: 자동화 단계
- **환경 구성**: dev/staging/prod 차이점
- **롤백 절차**: 문제 발생 시 복구 방법

### 8. 리스크 & 대안 (Risks & Alternatives)
- **알려진 리스크**: 발생 가능한 문제
- **대응 방안**: 각 리스크에 대한 완화 전략
- **고려했으나 선택하지 않은 대안**: 이유 포함

## 파일 저장
- 경로: `docs/SDD-{feature-name}.md`
- 네이밍: 기능 이름을 kebab-case로

## 원칙
- 코드를 작성하기 전에 항상 SDD를 먼저 작성
- 과도한 설계 지양: 현재 필요한 것에 집중
- 결정사항에는 반드시 근거를 포함
- 팀원이 이해할 수 있는 수준의 상세도 유지
