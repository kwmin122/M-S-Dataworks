## 한 줄 요약

Kira Bot을 "GO/NO-GO 분석 도구"에서 **"입찰 전 과정 자동화 플랫폼"**으로 확장 중. Phase 1 (A-lite 제안서 MVP + 회사 DB + 수정 학습 루프) **코드 구현 완료 + 배포 수준 코드 리뷰 통과**.

## 설계 문서 대비 진행 현황

### §2. 3계층 학습 모델

| 레이어 | 설계 | 현재 상태 |
|--------|------|-----------|
| **Layer 1 기본 모델** | 유튜브 50편 + 블로그 + 공식가이드 → LLM 2-Pass 정제 → ChromaDB | **코드 완성**. `knowledge_harvester.py` (Pass 1), `knowledge_dedup.py` (Pass 2), `knowledge_db.py` (ChromaDB). 소스 URL 큐레이션 완료 (`data/layer1_sources/` — YouTube 54편, 블로그 56편, 공식문서 21편). **실제 텍스트 수집+벡터화는 미실행** (사람이 큐레이션 확인 후 실행 필요). |
| **Layer 2 회사 맞춤** | 과거 제안서 업로드 → 문체/구조/강점 분석 → 회사 프로필 | **코드 완성**. `company_db.py` (실적/인력 ChromaDB), `company_analyzer.py` (문체 분석). `section_writer.py`에 Layer 2 프롬프트 주입 통합 완료. |
| **Layer 2 수정 Diff** | AI 초안 vs 사용자 수정 → 패턴 감지 → 자동 반영 | **코드 완성**. `diff_tracker.py` (diff 추출), `auto_learner.py` (1회=기록, 2회=후보, 3회+=자동반영). Edit Rate KPI 계산 포함. threading.Lock 스레드 안전. |
| **Layer 3 승패 분석** | 낙찰 vs 탈락 비교 → 승패 패턴 DB | **미구현** (Phase 2~3 범위) |

### §3. 8개 확장 모듈

| 모듈 | 설계 범위 | 현재 상태 |
|------|-----------|-----------|
| **A. 기술제안서 자동 생성** | RFP + Layer 1/2/3 → DOCX 100+ 페이지 | **A-lite 완성** (Layer 1+2 기반). `proposal_planner` → `section_writer` × N (병렬) → `quality_checker` → `document_assembler` (mistune AST + python-docx). API: `/api/generate-proposal-v2` |
| **B. 수행계획서/WBS** | 과업분석 → WBS → 간트차트 → 인력배치표 | **미구현** (Phase 2) |
| **C. PPT 발표자료** | 제안서 → 핵심 추출 → PPTX + 발표 노트 | **미구현** (Phase 2) |
| **D. 회사 역량 DB** | 실적/인력/과거 제안서 구조화 벡터 DB | **코드 완성**. `company_db.py`. 온보딩 UI는 미구현. |
| **E. 가격제안서** | SW기술자 노임단가 기반 원가 자동 산출 | **미구현** (Phase 3) |
| **F. 실적/경력 기술서** | RFP 별지 양식 감지 + 유사실적 자동 매칭 | **미구현** (Phase 2) |
| **G. 제출 체크리스트** | RFP → 제출서류 목록 추출 + 블라인드 검증 | **API 완성** (`checklist_extractor.py`, `/api/checklist`). 프론트엔드 UI 미구현. |
| **H. 승률 대시보드** | 입찰 이력 분석 + 경쟁사 패턴 + 추천 | **미구현** (Phase 3) |

### §5. 기술 아키텍처 확장

설계 문서의 Agentic 워크플로우 중 구현 완료된 부분:

```
document_parser → rfx_analyzer → matcher ← 기존 (동작 중)
                                    ↓
              proposal_planner (제안 아웃라인 생성) ← ✅ 구현
                  ↓
              section_writer × N (병렬 생성) ← ✅ 구현
                  ↓
              quality_checker (블라인드/모호 표현 감지) ← ✅ 구현
                  ↓
              document_assembler (DOCX 조립) ← ✅ 구현

미구현:
  wbs_generator, personnel_matcher, cost_calculator
```

## 현재 코드베이스 상태

### 새로 만든 파일 (rag_engine/, 15개 모듈)

```
rag_engine/
  llm_utils.py             ← LLM retry+timeout (call_with_retry, 지수 백오프 429/500/502/503)
  knowledge_models.py      ← 데이터 모델 (KnowledgeUnit, ProposalSection, ProposalOutline 등)
  knowledge_db.py          ← ChromaDB wrapper (proposal_knowledge collection, sha256 복합ID)
  knowledge_harvester.py   ← LLM Pass 1 지식 추출 (7카테고리)
  knowledge_dedup.py       ← LLM Pass 2 충돌 해소 (AGREE/CONDITIONAL/CONFLICT)
  proposal_planner.py      ← RFP → 섹션 아웃라인 (배점 비례 페이지 분배)
  section_writer.py        ← Layer 1+2 지식 주입 LLM 섹션 생성
  quality_checker.py       ← 블라인드 위반(한글 조사 인식) + 모호 표현 감지
  document_assembler.py    ← mistune 3.x AST + python-docx DOCX 조립
  proposal_orchestrator.py ← 전체 파이프라인 오케스트레이터 (ThreadPoolExecutor)
  company_db.py            ← 회사 역량 DB (실적/인력 ChromaDB)
  company_analyzer.py      ← 과거 제안서 문체/구조/강점 분석
  checklist_extractor.py   ← RFP → 제출서류 체크리스트 추출
  diff_tracker.py          ← AI vs 사용자 수정 diff 추출 + 패턴 키 해싱
  auto_learner.py          ← 수정 패턴 → Layer 2 자동 학습 (threading.Lock)
```

### 새로 추가된 API 엔드포인트

| 서버 | Method | Path | 설명 |
|------|--------|------|------|
| rag_engine :8001 | POST | `/api/generate-proposal-v2` | A-lite DOCX 제안서 생성 (Pydantic 검증) |
| rag_engine :8001 | POST | `/api/checklist` | 제출 체크리스트 추출 |
| rag_engine :8001 | POST | `/api/edit-feedback` | 사용자 수정 diff → 자동 학습 |
| legacy :8000 | POST | `/api/proposal/generate-v2` | → rag_engine 프록시 |

### 코드 품질 (배포 수준)

코드 리뷰에서 발견된 모든 CRITICAL (3건) + IMPORTANT (8건) 이슈 수정 완료:
- 서버 경로 노출 → `docx_filename`으로 교체
- 스레드 안전성 → `threading.Lock`
- 입력 검증 → Pydantic `RfxResultInput` 스키마
- LLM retry → `call_with_retry` 전 모듈 적용
- 마크다운 파싱 → mistune AST (정규식 제거)
- 한글 단어 경계 → 조사 인식 정규식
- 파일명 sanitization → 화이트리스트 정규식

## 즉시 해야 할 일 (우선순위)

### 1. Layer 1 데이터 실제 수집 + 벡터화 (사람 작업 필요)
`data/layer1_sources/` JSON의 URL들을 실제로 수집해야 함:
- YouTube: `youtube-transcript-api`로 자막 추출 → `knowledge_harvester` 실행
- 블로그: `trafilatura`로 본문 추출 → 같은 파이프라인
- 공식 문서: PDF/HWP → `document_parser` → 같은 파이프라인
- **사전 작업**: 사람이 URL 큐레이션 품질 확인 (S/A/B/C 등급 부여)

### 2. Golden Test 10건
실제 나라장터 공고 10건 선정 → A-lite 제안서 생성 → 품질 평가
목표: 커버리지 30%+, 치명적 실수 0건

### 3. 프론트엔드 UI 미완성 부분
- 제안서 DOCX 다운로드 링크 + 섹션 미리보기 ContextPanel
- 체크리스트 패널
- 회사 DB 온보딩 위저드

### 4. auto_learner 영속성
인메모리 → 서버 재시작 시 학습 데이터 사라짐. FastAPI lifespan 이벤트에서 `save_state()`/`load_state()` 호출 필요.

## 우리의 목표 (핵심)

> **"입찰 공고가 나오면, Kira Bot이 알아서 분석하고, 제안서를 쓰고, PPT를 만들고,
> 부족한 서류를 알려주고, 제출 마감을 관리하고, 결과를 추적하여 매번 더 똑똑해지는
> — 기업의 AI BD 팀이 되는 것."**

### 현재 도달 지점
```
공고 검색 → RFP 분석 → GO/NO-GO → 제안서 초안(A-lite DOCX) → 체크리스트(API)
                                         ↑ Layer 1+2 지식 주입
                                         ↑ 사용자 수정 자동 학습
```

### 아직 가야 할 길
```
수행계획서/WBS → PPT → 실적기술서 → 가격제안서 → 승률 예측 → 포트폴리오 전략
Layer 3 승패 분석 → 발주처 프로파일링 → Post-Bid 자동 학습
민간 입찰 개방형 → 정부지원사업 사업계획서
```

### 품질 기준 (반드시 지킬 것)
- **배포 수준 코딩** — MVP가 아님. 에러 핸들링, 입력 검증, 스레드 안전성, LLM retry 모두 필수.
- 기존 `/api/proposal/generate` 엔드포인트 **절대 삭제 금지** — 기존 챗봇 플로우 핵심.
- `CLAUDE.md`에 정의된 모든 코딩 규칙 준수.
