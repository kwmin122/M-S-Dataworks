> 2026-02-27 | 민경욱

## 1. 현재 위치 — 우리가 이미 가진 것

```
공고 발견 ──▶ 자격 분석 ──▶ GO/NO-GO 판정 ──▶ Q&A 채팅
  ✅ 완성       ✅ 완성        ✅ 완성           ✅ 완성
```

Kira Bot은 **입찰 의사결정의 앞단**을 이미 해결했다. 하지만 기업이 "GO"를 결정한 후에 실제로 해야 할 일은 훨씬 더 많다.

## 3. 비전 — "입찰의 전체 라이프사이클을 자동화한다"

현재 Kira Bot이 커버하는 영역과 확장 가능한 영역을 맵핑하면:

```
입찰 라이프사이클
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase 1: 발견          Phase 2: 분석         Phase 3: 준비           Phase 4: 제출
┌──────────────┐  ┌──────────────┐  ┌────────────────────┐  ┌──────────────┐
│ 공고 수집     │  │ RFP 분석      │  │ 제안서 초안 작성     │  │ 최종 검토     │
│ 관심 필터링   │  │ 자격요건 추출  │  │ 수행계획서 생성      │  │ 제출 체크리스트│
│ 일괄 평가     │  │ GO/NO-GO     │  │ PPT 발표자료 생성    │  │ 가격 입찰 지원│
│              │  │ Q&A 채팅      │  │ 실적기술서 정리      │  │ 제출 알림     │
└──────────────┘  └──────────────┘  │ 투입인력 구성표      │  └──────────────┘
    ✅ 완성           ✅ 완성        │ 가격제안서 산출      │      🔮 미래
                                    │ 사업계획서 작성      │
                                    └────────────────────┘
                                         🚀 핵심 확장 영역
```

## 4. 확장 아이디어 — 8개 모듈 브레인스토밍

### 모듈 A: 📝 기술제안서 자동 생성기 (핵심 킬러 기능)

**문제**: 기술제안서는 보통 50~200페이지. 평가항목(사업이해도, 기술적 접근, 수행방법론, 인력구성, 품질관리)에 맞춰 써야 하는데, 매번 처음부터 쓴다.

**해결 방향**:
```
RFP 분석 결과 (이미 있음)
    + 회사 실적 DB (새로 구축)
    + 제안서 템플릿 라이브러리 (새로 구축)
    + 평가기준 매핑 엔진 (새로 구축)
    ──────────────────────
    = 제안서 초안 자동 생성 (DOCX)
```

**구현 아이디어**:
- RFP에서 추출한 평가항목별로 섹션 자동 구성
- 조달청 협상계약 평가기준 (기술 70점 + 가격 30점) 매핑
  - 제안개요 (사업이해도, 추진전략)
  - 업체역량 (유사사업 실적, 기술인력)
  - 사업전략 (기술적 접근, 시스템 구성)
  - 사업관리 (일정관리, 위험관리)
  - 품질관리 (품질보증, 산출물)
- 회사의 과거 제안서를 학습해서 "우리 회사 톤"으로 생성
- **Loopio/DeepRFP 스타일**: 과거 제안서 라이브러리에서 유사 답변을 찾아 재사용
- 출력: 평가항목에 맞춰 구조화된 Word(.docx) 문서

**차별점 (vs 클라이원트/독스헌트)**:
- 우리는 이미 **RFP 분석 → 자격요건 추출 → 평가기준 파악**이 되어 있음
- 제안서 생성이 GO/NO-GO 판정과 자연스럽게 연결됨
- "이 공고 GO야 → 그럼 제안서 초안 만들어줘" 원클릭 워크플로우

---

### 모듈 B: 📊 수행계획서 / WBS 생성기

**문제**: 사업수행계획서(일정, WBS, 마일스톤, 투입인력)는 모든 기술제안서의 필수 구성요소. 프로젝트 관리 노하우가 필요한 영역.

**해결 방향**:
```
RFP 과업내용 분석
    + 사업기간/규모 추출
    + 유사 프로젝트 패턴 DB
    ──────────────────────
    = WBS 자동 생성 + 간트차트 + 인력배치표
```

**구현 아이디어**:
- RFP에서 과업범위, 산출물 목록, 사업기간을 추출 (rfx_analyzer 확장)
- 표준 SW개발 방법론(폭포수/애자일/반복점진적) 템플릿 적용
- WBS 분해 → 일정 자동 산출 → 간트차트 이미지 생성
- 투입인력 구성표 (PM, PL, 개발자, QA 등) 자동 배치
- 출력: 엑셀(WBS) + 이미지(간트차트) + Word 섹션 삽입

---

### 모듈 C: 🎯 PPT 발표자료 자동 생성기

**문제**: 기술평가 발표는 보통 20~30분. 핵심을 추려서 PPT를 만들어야 하는데, 제안서 200페이지를 20장으로 압축하는 것 자체가 큰 작업.

**해결 방향**:
```
생성된 제안서 초안 (모듈 A)
    + 평가항목별 핵심 메시지 추출
    + PPT 템플릿 (회사 CI 반영)
    ──────────────────────
    = 발표용 PPTX 자동 생성
```

**구현 아이디어**:
- 제안서 각 섹션에서 "평가위원이 볼 핵심 3줄"을 추출
- 슬라이드 구성: 표지 → 사업이해 → 추진전략 → 시스템구성도 → 일정 → 인력 → Q&A
- 조달청 평가기준 배점 비중에 따라 슬라이드 수 자동 배분
- 다이어그램/아키텍처 도식 자동 생성 (Mermaid → 이미지)
- 회사 로고, CI 컬러 자동 적용
- 출력: PPTX (python-pptx 또는 기존 pptx 스킬 활용)

---

### 모듈 D: 🏢 회사 역량 DB (Company Knowledge Base)

**문제**: 매번 제안서를 쓸 때마다 "우리 회사 실적이 뭐였지?", "인력 경력기술서 어디 있지?" 반복.

**해결 방향**:
```
회사 기본정보 (사업자등록, 면허, 인증서, 자본금)
    + 수행실적 DB (프로젝트명, 기간, 금액, 고객사, 핵심기술)
    + 인력 Pool (이력서, 자격증, 경력사항)
    + 과거 제안서 라이브러리
    + 회사 소개서 / 홍보자료
    ──────────────────────
    = 구조화된 회사 역량 벡터 DB
```

**이것이 모든 모듈의 기반이 되는 이유**:
- 모듈 A (제안서): "유사 실적 3건을 실적기술서 양식으로"
- 모듈 B (수행계획서): "PM 후보 3명의 경력 자동 매핑"
- 모듈 C (PPT): "회사 CI + 대표 프로젝트 사진 삽입"
- GO/NO-GO (기존): 더 정확한 자격요건 매칭 (현재보다 훨씬 풍부한 데이터)

**구현 아이디어**:
- 온보딩 위저드: 사업자등록증 OCR → 자동 입력
- 실적 업로드: 계약서/실적증명서 PDF 업로드 → AI 구조화
- 인력 관리: 이력서 업로드 → 경력/자격증 자동 추출
- 벡터 검색: "클라우드 마이그레이션 관련 유사 실적" 시맨틱 검색

---

### 모듈 E: 💰 가격제안서 / 원가산출 보조

**문제**: 가격 평가(30점)도 당락을 좌우함. 인건비 산출(SW기술자 등급별 단가), 경비, 이윤 계산이 복잡.

**해결 방향**:
- 한국소프트웨어산업협회 SW기술자 노임단가표 내장
- RFP 추정가격 + 사업기간 + 투입인력 → 원가 자동 산출
- 적정 할인율 시뮬레이션 (추정가격 대비 투찰률)
- 출력: 엑셀 원가산출서 (.xlsx)

---

### 모듈 F: 📋 실적기술서 / 경력기술서 자동 생성

**문제**: 투입인력 경력기술서, 유사실적기술서는 매번 양식에 맞춰 다시 정리해야 함.

**해결 방향**:
```
회사 역량 DB (모듈 D)
    + RFP 요구 양식 감지
    + 표준 양식 라이브러리 (조달청, 기관별)
    ──────────────────────
    = 맞춤형 실적기술서 / 경력기술서 (DOCX)
```

**구현 아이디어**:
- RFP에서 "별지 양식" 감지 → 해당 양식에 맞춰 자동 채움
- 유사실적을 "관련도 순"으로 정렬 (RFP 과업 vs 실적 내용 유사도)
- 인력 배치 최적화: 평가 가점 극대화를 위한 인력 조합 추천

---

### 모듈 G: ✅ 제출 체크리스트 / 서류 검증기

**문제**: 입찰 무효 사유 1위가 "서류 누락". 제출서류 목록을 빠뜨리면 아무리 좋은 제안서도 소용없음.

**해결 방향**:
- RFP에서 제출서류 목록 자동 추출 (이미 자격요건 추출 파이프라인 존재)
- 체크리스트 생성 → 사용자가 하나씩 체크
- 서류 업로드 시 파일명/형식 자동 검증
- 마감일 D-day 알림 (이메일/카카오)

---

### 모듈 H: 📈 입찰 이력 분석 / 승률 대시보드

**문제**: "우리 회사가 어떤 유형의 입찰에 강한지" 데이터 기반 인사이트가 없음.

**해결 방향**:
- 입찰 참여 이력 DB (참여/낙찰/유찰)
- 승률 분석: 사업 유형별, 금액대별, 발주처별
- 경쟁사 낙찰 패턴 분석 (공공데이터 기반)
- 추천 엔진: "이번 달 승률 높은 공고 Top 10"
- 시각화 대시보드 (React 또는 내장 차트)

## 6. 기술 아키텍처 확장 방향

### 현재 → 확장 아키텍처

```
현재 파이프라인:
  document_parser → rfx_analyzer → matcher → chat_router

확장 파이프라인 (Agentic 멀티스텝):
  document_parser
      → rfx_analyzer (요건 추출 + 평가기준 매핑)
      → matcher (GO/NO-GO)
      → proposal_planner (제안 전략 수립 에이전트)
          → section_writer (섹션별 초안 작성 에이전트)
          → wbs_generator (WBS/일정 생성 에이전트)
          → personnel_matcher (인력 매칭 에이전트)
          → cost_calculator (원가 산출 에이전트)
      → document_assembler (DOCX/PPTX/XLSX 조립)
      → quality_checker (품질 검증 에이전트)
```

### Agentic 워크플로우 설계

Deloitte/Stack AI의 2026 Agentic AI 가이드에 따르면, 이런 파이프라인은 **"Sequential Pipeline + Hierarchical Supervisor"** 패턴이 적합:

```
Supervisor Agent (proposal_planner)
    ├── Worker: section_writer × N (병렬, 섹션별)
    ├── Worker: wbs_generator (순차, 과업 분석 후)
    ├── Worker: personnel_matcher (순차, 인력 DB 조회)
    ├── Worker: cost_calculator (순차, 인력 배치 후)
    └── Assembler: document_assembler (최종 조립)

각 Worker는:
    1. 공유 메모리(RFP 분석 결과, 회사 DB)에 접근
    2. 자기 영역의 산출물 생성
    3. 결과를 공유 메모리에 저장
    4. Supervisor가 품질 검증 후 다음 단계로
```

## 12. 핵심 추가 전략 — "먼저 배우고, 그다음 생성한다"

> **철학**: 공공기관이 "이거 해줄래?" → 기업이 "내가 할게!" 하고 낙찰받는 구조.
> 공공기관은 까다롭다. AI가 제안서를 쓰려면, **사람 전문가가 수십 년간 쌓은 노하우를 먼저 학습**해야 한다.

### 12-1. 2계층 학습 모델 — "기본 모델 + 회사 맞춤 모델"

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 기본 모델 (Base Model)                      │
│  ─ 우리가 구축하는 "제안서 작성 전문가 AI"              │
│  ─ 유튜브 50~N편 + 블로그 + 공식 가이드 학습           │
│  ─ 모든 사용자에게 공통 적용                           │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │  Layer 2: 회사 맞춤 모델 (Custom Model)           │ │
│  │  ─ "학습시킬 제안서를 업로드해주세요"               │ │
│  │  ─ 과거 제안서 → 회사 고유 톤/구조/전략 학습       │ │
│  │  ─ 회사마다 다른 템플릿, 용어, 강조점 반영         │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

**사용자 경험 플로우**:
```
1️⃣ 처음 가입 시:
   "과거에 작성한 제안서가 있으시면 업로드해주세요.
    Kira가 귀사의 제안 스타일을 학습합니다."
   → PDF/DOCX/HWP 업로드 (다수 가능)
   → AI가 분석: 문체, 구조, 반복 패턴, 강점 표현 방식

2️⃣ 제안서 생성 시:
   기본 모델 (전문가 노하우) + 회사 맞춤 모델 (톤/구조)
   = "업계 베스트 프랙티스 + 우리 회사 색깔"의 융합
```

### 12-3. Layer 1에 녹여야 할 핵심 노하우 (리서치 종합)

서치 결과를 종합하면, AI가 반드시 학습해야 할 "제안서의 룰"은:

#### A. 평가위원의 현실 이해 ("평가의 역설")
- 평가위원은 제안서를 **자세히 읽지 않는다** (시간이 없다)
- 보는 것: **그래픽, 캡션, 제목, 강조 문구, Executive Summary**
- → AI 생성 시: 모든 페이지에 "이 페이지의 핵심 1줄" 캡션 자동 생성

#### B. 배경-목적-비전의 주체 구분
- 배경 = **최종고객** 입장 ("시장이 이렇게 변하고 있어")
- 목적 = **발주사 실행부서** 입장 ("이 사업으로 이걸 해결하고 싶어")
- 비전 = **제안사** 입장 ("우리가 이렇게 만들어 드리겠습니다")
- → AI 생성 시: 각 섹션의 "화자"를 자동으로 구분하여 톤 조절

#### C. 논리 구조 = "제안 나무(Recommendation Tree)"
- 최상위: 핵심 주제
- 하위: 뒷받침 근거가 나뭇가지처럼 연결
- → AI 생성 시: 모든 주장에 "근거 → 수치 → 출처/진척도" 구조 강제

#### D. 감점 회피 체크리스트
- 블라인드 평가 위반 (회사명/로고 노출) → **자동 제거 기능**
- 설명 없는 표/차트 → **표마다 해설 문단 자동 생성**
- 근거 없는 추상적 주장 → **"최고 수준" 같은 표현 경고**
- RFP 규격 위반 → **페이지 수, 폰트, 용지 사이즈 자동 검증**
- 정책 방향 불일치 → **2026 정부 정책 키워드 자동 삽입**

#### E. 정부지원사업 사업계획서 특화 (PSST 프레임워크)
- **Problem**: 구체적 시장 데이터 + 실제 고객 인용
- **Solution**: 기술적 차별점 + 프로토타입/MVP 존재 여부
- **Scale-up**: TAM→SAM→SOM 시장 규모 + 구체적 시점별 계획
- **Team**: 핵심 인력의 "이 문제를 풀 수 있는" 직접 경험
- → AI 생성 시: PSST 각 항목별 템플릿 + 자동 웹서치로 시장 데이터 삽입

#### F. PPT 발표 전략
- A3 가로형이 일반적 (공공기관)
- 2~4단 분할 레이아웃
- 그래픽 50% 미만 유지 (과하면 역효과)
- 컬러 패널로 핵심 수치 강조
- "사례로 증명" — 유사 실적을 스토리텔링
- → AI 생성 시: 발표 시나리오(20분/30분) 자동 구성 + 슬라이드별 발표 노트

### 12-4. Layer 2 구축 — 회사별 맞춤 학습

#### 사용자 온보딩 플로우

```
Step 1: "과거 제안서 업로드" (낙찰된 것 우선)
    → 3~10건 권장
    → PDF/DOCX/HWP 자동 파싱

Step 2: AI 자동 분석 (뒤에서 돌아감)
    → 문체 분석: 어휘 수준, 문장 길이, 존칭 패턴
    → 구조 분석: 섹션 구성, 목차 패턴, 페이지 배분
    → 강점 표현: "우리 회사가 반복적으로 강조하는 것" 추출
    → 시각화 스타일: 표/그래프 사용 패턴, CI 컬러
    → 용어 사전: 회사 특유의 약어, 프로젝트명, 솔루션명

Step 3: 맞춤 프로필 생성
    → "귀사의 제안서 스타일을 분석했습니다:
        - 문체: 격식체, 평균 문장 28자
        - 강점: 인력 경험 강조, 기술 아키텍처 도식 활용
        - 구조: 사업이해(15%) → 기술접근(40%) → 관리(20%) → 실적(25%)
        - 특징: 비유/은유 활용, 정책 키워드 적극 반영"

Step 4: 생성 시 적용
    → 기본 모델의 노하우 + 회사 맞춤 스타일 = 최종 출력
```

#### 기술적 구현: RAG + 프롬프트 앙상블

```
[기본 모델 프롬프트]
  "당신은 공공조달 제안서 작성 전문가입니다.
   다음 규칙을 반드시 따르세요: {Layer 1 노하우 DB에서 검색된 규칙들}"

+ [회사 맞춤 프롬프트]
  "이 회사의 제안서 스타일: {Layer 2 분석 결과}
   과거 유사 제안서에서 이런 표현을 사용했습니다: {RAG 검색 결과}"

+ [RFP 컨텍스트]
  "현재 RFP 분석 결과: {rfx_analyzer 출력}
   평가항목: {추출된 배점표}
   요구사항: {추출된 과업 범위}"

= 최종 제안서 섹션 생성
```

**왜 Fine-tuning이 아닌 RAG인가?**:
- 회사가 새 제안서를 쓸 때마다 DB 업데이트만 하면 됨 (재학습 불필요)
- 비용 1/10 (Fine-tuning GPU 비용 vs RAG 벡터 DB)
- 출처 투명성 (어떤 과거 제안서에서 참조했는지 보여줄 수 있음)
- 나중에 품질 더 올리고 싶으면 Fine-tuning 추가 가능 (단계적 접근)

---

### 12-5. 정부지원사업 사업계획서 영역 확장

공공조달 입찰 제안서만이 아니라, **정부지원사업 사업계획서**도 같은 파이프라인으로 커버:

| 대상 사업 | 작성 프레임워크 |
|----------|--------------|
| 예비창업패키지 | PSST (Problem-Solution-ScaleUp-Team) |
| 초기창업패키지 | 사업모델, 시장분석, 실행계획, 팀 구성 |
| TIPS (민간투자연계) | 기술성, 사업성, 팀, 투자유치 계획 |
| AI 바우처 | AI 기술 활용 계획, 기대효과, 데이터 계획 |
| 스마트공장 | 도입 목적, AS-IS/TO-BE, 투자계획, ROI |

→ 사업유형별 템플릿 + 과거 합격 사례 학습 = **사업계획서 자동 생성기**

---

### 12-6. 학습 파이프라인 구축 로드맵

```
Week 1-2: 유튜브 50편 큐레이션 + 트랜스크립트 수집
Week 2-3: 블로그/공식 가이드 크롤링 + 구조화
Week 3-4: 노하우 DB 구축 (카테고리별 벡터 DB)
Week 4-5: 기본 모델 프롬프트 엔지니어링 + 테스트
Week 5-6: 회사 맞춤 온보딩 파이프라인 구축
Week 6-8: 실제 RFP로 제안서 생성 테스트 + 품질 평가
```

**성공 지표**:
- 전문가가 초안 리뷰 시 "70% 이상 그대로 사용 가능" → 성공
- 생성 시간: RFP 업로드 후 **10분 이내** 50페이지 초안 완성
- 감점 요소 0건 (블라인드 위반, 규격 위반 등 자동 검증)
