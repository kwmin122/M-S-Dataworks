# 제안요청서(RFP) 자동 분석 RAG 시스템 - 클로드 코드 전달용 프롬프트

---

## 프로젝트 목표

다양한 한국 공공/민간 제안요청서(RFP) 문서를 업로드하면:
1. **1단계**: 핵심 정보를 구조화 요약하여 즉시 제공
2. **2단계**: RAG 기반 Q&A로 문서 내 상세 질의응답

> 핵심: CCTV, IT, 건설, 컨설팅 등 **어떤 분야의 RFP가 와도** 동일한 프레임워크로 요약 가능해야 함

---

## 시스템 아키텍처

```
[문서 업로드] → [파싱(PDF/HWP/DOCX)] → [전문 텍스트 추출]
                                              ↓
                                    ┌─────────┴─────────┐
                                    ↓                     ↓
                            [1단계: 구조화 요약]    [청킹 + 임베딩]
                            (LLM이 스키마 기반       (벡터DB 저장)
                             정보 추출)                   ↓
                                    ↓               [2단계: RAG Q&A]
                            [사용자에게 요약 출력]   (검색 → LLM 응답)
```

## 핵심 설계 포인트 (브레인스토밍)

### 1. 왜 3섹션인가?

경욱님이 정의한 3섹션 구조는 입찰 실무의 의사결정 흐름과 정확히 일치:

```
1. 사업유형     → "이 사업에 참여할까?" (Go/No-Go 결정)
2. 입찰자격조건  → "우리가 자격이 되나?" (자격 검증)
3. 평가방법     → "이길 수 있나? 전략은?" (경쟁력 분석)
```

### 2. "추가로 보면 좋은 사항"의 중요성

분리발주, 별도발주, 기존시스템 연동 정보는 실무에서 가격 산정과 파트너링 전략에 직결:
- 분리발주 → 해당 품목을 우리가 직접 공급해야 하는지, 아니면 발주처가 별도로 사는지
- 별도발주(직접구매) → 이 부분은 우리 견적에서 빼야 함
- 기존 시스템 → 특정 제조사 제품 Lock-in이 있는지 (경쟁 제한 요소)

### 3. 정량적 평가 기준의 "그대로 출력" 원칙

정량적 평가 기준표는 **가공하지 않고 원문 그대로** 출력해야 함:
- 실무자가 자기 회사 데이터를 대입해서 점수를 직접 계산해야 하므로
- 요약하면 오히려 정보 손실 → 잘못된 점수 계산 위험

### 4. 범용성 확보 전략

다양한 RFP에 대응하기 위한 핵심 전략:

| 도전과제 | 해결 방법 |
|----------|-----------|
| RFP마다 용어가 다름 | 동의어 매핑 (사업비=계약금액=총사업비) |
| 구조가 다름 | 의미 기반 추출 (키워드 패턴 + LLM 이해력) |
| 분야마다 "필요한 것"이 다름 | 문서 유형 자동 감지 → 포커스 조정 |
| 없는 항목 | "문서에 명시되지 않음"으로 명시 (빈칸X) |
| 정보가 분산됨 | 여러 장/절에서 동일 주제 정보를 모아서 통합 |

### 5. 향후 확장 아이디어

- **자격조건 자동 체크**: 회사 프로필 저장 → RFP 올리면 자동으로 Go/No-Go 판정 - 구현하자 이것도. 
- **정량적 점수 자동 계산**: 회사 정보(신용등급, 실적, 인력) 입력하면 예상 점수 산출
- **경쟁사 분석**: 과거 낙찰 데이터와 비교하여 경쟁력 분석   
- **제안서 목차 자동 생성**: 평가기준표의 고배점 항목 중심으로 제안서 골격 자동 생성
- **마감일 트래커**: RFP에서 추출한 일정을 캘린더에 자동 등록
- **유사 RFP 비교**: 이전에 분석한 RFP들과 비교하여 차이점 하이라이트
