## §1. 비전과 핵심 철학

### 비전

> "입찰 공고가 나오면, Kira Bot이 알아서 분석하고, 제안서를 쓰고, PPT를 만들고,
> 부족한 서류를 알려주고, 제출 마감을 관리하는 — 기업의 AI BD 팀이 되는 것."

### 한 줄 요약

"이 공고, 우리가 할 수 있어?" → **"이 공고, 제안서까지 다 만들어줘"**

### 핵심 철학: "먼저 배우고, 그다음 생성한다"

공공기관이 "이거 해줄래?" → 기업이 "내가 할게!" 하고 낙찰받는 구조.
공공기관은 까다롭다. AI가 제안서를 쓰려면, 사람 전문가가 수십 년간 쌓은 노하우를 먼저 학습해야 한다.

## §2. 3계층 학습 모델 — 차별화의 핵심

```
┌───────────────────────────────────────────────────────────┐
│  Layer 1: 기본 모델 (Base Model)                            │
│  ─ 우리가 구축하는 "제안서 작성 전문가 AI"                   │
│  ─ 유튜브 50~N편 + 블로그 + 공식 가이드 학습                │
│  ─ 모든 사용자에게 공통 적용                                 │
│                                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Layer 2: 회사 맞춤 모델 (Custom Model)               │   │
│  │  ─ "학습시킬 제안서를 업로드해주세요"                   │   │
│  │  ─ 과거 제안서 → 회사 고유 톤/구조/전략 학습           │   │
│  │  ─ 회사마다 다른 템플릿, 용어, 강조점 반영             │   │
│  │                                                      │   │
│  │  ┌───────────────────────────────────────────────┐   │   │
│  │  │  Layer 3: 승패 분석 모델 (Win-Loss Model)       │   │   │
│  │  │  ─ 낙찰 vs 탈락 제안서 비교 학습                │   │   │
│  │  │  ─ "이 회사가 이길 때와 질 때의 차이" 패턴 추출 │   │   │
│  │  │  ─ 경쟁사 낙찰 제안서 (정보공개 청구 가이드)    │   │   │
│  │  │  ─ 사용할수록 데이터 쌓이는 플라이휠            │   │   │
│  │  └───────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────────┘
```

#### Step 2: 원시 텍스트 → 구조화 지식 변환 (Knowledge Distillation)

유튜브 자막 원문을 그대로 RAG에 넣으면 "어 그래서 이게 뭐냐면요 ㅎㅎ" 이런 게 검색된다. **LLM 2-Pass 정제 파이프라인**이 핵심.

**Pass 1 — Extraction (원문 → 지식 단위)**

프롬프트:
```
당신은 공공조달 제안서 전문가입니다.
아래 텍스트에서 제안서 작성에 도움이 되는 지식을 추출하세요.

각 지식 단위를 다음 JSON 형식으로 출력:
{
  "category": "structure|evaluation|writing|visual|strategy|compliance|pitfall",
  "subcategory": "...",
  "rule": "한 문장으로 된 핵심 규칙",
  "explanation": "왜 이 규칙이 중요한지 2-3문장",
  "example_good": "잘 쓴 예시 (있으면)",
  "example_bad": "못 쓴 예시 (있으면)",
  "source_type": "youtube|official_guide|blog|textbook",
  "confidence": 0.0~1.0
}
```

7개 카테고리 분류 체계:

| 카테고리 | 내용 | 예시 |
|----------|------|------|
| **structure** | 문서 구조/목차 | "기술제안서는 이해도→기술성→관리방안 순서" |
| **evaluation** | 평가기준/배점 | "기술성 배점 60점 중 핵심은 구현방안 30점" |
| **writing** | 작성 기법 | "한 페이지에 핵심 메시지 1개만" |
| **visual** | 시각화/레이아웃 | "개조식 금지, 도표+다이어그램 70% 이상" |
| **strategy** | 전략적 판단 | "발주처 OO 유형은 가격보다 실적 중시" |
| **compliance** | 규정/자격 | "직접생산확인증명서 업종코드 정확히 일치 필수" |
| **pitfall** | 흔한 실수 | "제안요청서 7페이지 '별도 제출' 문구 놓치면 탈락" |

**Pass 2 — Deduplication & Conflict Resolution (중복 제거 + 모순 해소)**

같은 내용이 다른 소스에서 반복 → confidence 상향 + 소스 병합. **모순 해소는 3단계 파이프라인**으로 처리:

```
for chunk in extracted_knowledge:
    similar = vector_search(chunk.rule, top_k=5, threshold=0.92)
    if similar:
        if is_agreeing(chunk, similar[0]):
            merge_knowledge(existing=similar[0], new=chunk)  # Case 1: 동의 → 소스 병합, confidence ↑
        elif is_conditional(chunk, similar[0]):
            split_conditional(chunk, similar[0])              # Case 2: 조건부 분기 → 둘 다 저장
        else:
            resolve_conflict(chunk, similar[0])               # Case 3: 진짜 충돌 → 가중 투표
    else:
        insert_knowledge(chunk)  # 신규 → 벡터DB 삽입
```

##### 모순 해소 3단계

**Case 1: 동의 (70% 예상)** — 같은 규칙, 다른 소스
```
YouTube A: "한 페이지에 핵심 메시지 1개" (confidence 0.55)
Blog B: "페이지당 하나의 메시지만" (confidence 0.35)
→ 병합: source_count=2, confidence = min(0.55 × 1.2, 0.95) = 0.66
```

**Case 2: 조건부 분기 (80% of "모순", ~24% 전체)** — 맥락이 다른 규칙
```
YouTube 강사: "실적기술서는 최대한 많이 넣어라"
조달청 가이드: "실적기술서는 상위 5건만 선별"
→ LLM 판정: "적격심사(가격위주)면 양 많이, 기술평가(배점위주)면 선별"
→ 둘 다 저장, condition 필드 추가:
  Rule A: "실적은 가능한 많이 제출" → condition: "적격심사 방식 입찰일 때"
  Rule B: "실적은 상위 5건만 선별" → condition: "기술평가 방식 입찰일 때"
```

**Case 3: 진짜 충돌 (10~20% of "모순", ~3~6% 전체)** — 같은 맥락, 다른 결론
```
블로그: "표지에 사업명만 적는다" (grade B, confidence 0.35)
공식가이드: "표지에 사업명+제안사명+제출일자" (grade S, confidence 0.85)
→ 가중 투표: 공식가이드 승
→ 승리 규칙: confidence 유지 + has_conflict_flag=true
→ 패배 규칙: confidence × 0.3 + deprecated_by 링크 (삭제하지 않음 — 법령 변경 시 복원 가능)
```

##### Pass 2 LLM 프롬프트 (충돌 감지용)

```
두 지식 단위가 모순인지 판별하세요:

Rule A: "{rule_a}" (소스: {source_a}, grade: {grade_a})
Rule B: "{rule_b}" (소스: {source_b}, grade: {grade_b})

다음 중 하나로 답하세요:
1. AGREE — 같은 내용을 다르게 표현 (→ 병합)
2. CONDITIONAL — 둘 다 맞지만 적용 조건이 다름 (→ 각각의 condition 설명)
3. CONFLICT — 같은 상황에서 다른 결론 (→ 어느 쪽이 더 정확한지 근거 제시)

JSON 형식:
{
  "verdict": "AGREE|CONDITIONAL|CONFLICT",
  "condition_a": "Rule A가 적용되는 조건 (CONDITIONAL일 때)",
  "condition_b": "Rule B가 적용되는 조건 (CONDITIONAL일 때)",
  "winner": "A|B (CONFLICT일 때)",
  "reasoning": "판단 근거"
}
```

##### Conflict Audit Log

모든 충돌 해소 과정을 `ConflictResolution` 레코드로 별도 저장:
```python
@dataclass
class ConflictResolution:
    rule_a_id: str
    rule_b_id: str
    verdict: str          # AGREE | CONDITIONAL | CONFLICT
    condition_a: str      # Case 2용
    condition_b: str      # Case 2용
    winner: str           # Case 3용
    reasoning: str
    resolved_at: str      # ISO timestamp
```
- UI에서 "이 규칙은 일부 소스에서 다른 의견이 있습니다" 표시 가능
- 품질 관리자가 자동 해소 결과를 사후 검토할 수 있는 감사 추적

##### Context Embedding for Conditional Rules

`condition` 필드가 있는 규칙은 임베딩 시 조건도 포함:
```
임베딩 텍스트 = rule + " [조건: " + condition + "]"
```
이렇게 하면 RFP에서 "기술평가" 키워드가 나올 때 기술평가 조건부 규칙이 더 높은 유사도로 검색됨. 조건 없는 보편 규칙은 항상 검색됨.

**예상 결과: 원시 100만+ 토큰 → 정제된 지식 단위 800~1,500개** (조건부 분기로 인해 일부 중복 허용 → 실제 900~1,800개 가능)

#### Step 2-A: Source-Weighted Confidence — 소스 신뢰도 가중 점수

단일 confidence 값이 아닌, **소스 유형별 기본 점수 × 교차검증 배수** 공식:

```
base_confidence:
  official_guide:      0.85   # 조달청/법제처 공식 문서
  evaluator_youtube:   0.75   # 평가위원 출신 강사
  winner_story:        0.70   # 낙찰 경험자 실전 후기
  consultant:          0.55   # 컨설턴트 방법론
  blog:                0.35   # 일반 블로그

cross_validation_multiplier:
  1개 소스만:    ×1.0
  2개 소스 일치: ×1.2
  3개+ 소스 일치: ×1.4 (cap at 0.95)

최종 confidence = min(base × multiplier, 0.95)
```

제안서 생성 시 **confidence 순 정렬**로 토큰 제한 내 최고 품질 지식만 주입.

모순 해결 (상세: 위 Pass 2 "모순 해소 3단계" 참조):
```
유튜브 강사: "실적기술서는 최대한 많이 넣어라" (confidence 0.55)
조달청 가이드: "실적기술서는 상위 5건만 선별" (confidence 0.85)
→ LLM 판정: CONDITIONAL (적격심사 vs 기술평가 맥락 차이)
→ 둘 다 저장 + condition 필드로 구분
→ 진짜 충돌(CONFLICT)인 경우에만 가중 투표로 승자 결정
```

#### Step 2-B: Temporal Versioning — 지식 유효기간 관리

공공조달 법령/기준은 자주 바뀐다. 지식에 유효기간을 부여:

```
knowledge_unit:
  rule: "기술성 배점은 70점 만점"
  valid_from: "2024-01"
  valid_until: null        # null = 현재 유효
  superseded_by: null
  freshness_score: 1.0     # 시간이 지나면 감소

freshness 계산:
  6개월 이내:    1.0
  6~12개월:     0.9
  1~2년:        0.7
  2년+:         0.5
  법령 기반:     항상 1.0 (superseded 될 때까지)
```

자동 갱신 트리거: 조달청 고시 업데이트 모니터링 → 관련 knowledge_unit에 `superseded_by` 태그 → 새 규칙으로 교체. "작년 기준으로 제안서 쓰는" 치명적 실수 방지.

#### Step 3: 벡터DB 구축 (Knowledge Indexing) — 2-Collection 구조

```
Collection 1: "proposal_knowledge"
  — 정제된 지식 단위 (Pass 2 결과, 800~1,500개)
  — 메타데이터: category, subcategory, source_type, confidence, freshness
  — 청킹: 이미 단위별로 분리 → 추가 청킹 불필요
  — 임베딩: text-embedding-3-small (비용 효율) 또는 3-large (정확도)

Collection 2: "proposal_templates"
  — 구조화된 템플릿/목차/체크리스트
  — 공종별, 사업유형별 분류
  — 청킹: 섹션 단위
```

검색 시나리오:
1. RFP에서 평가기준 추출 (기존 rfx_analyzer)
2. 평가기준 각 항목으로 proposal_knowledge 검색
3. "기술성 > 구현방안" → 관련 writing/visual/evaluation 지식 10~20개 retrieve
4. confidence × freshness 순 정렬 → 상위 N개를 제안서 생성 프롬프트에 주입

#### Step 4: 생성 시 지식 주입 방식 (Prompt Assembly)

```
[System Prompt]
당신은 대한민국 공공조달 제안서 전문가입니다.

[Layer 1 — 검색된 보편 지식]
## 이 유형의 제안서에 적용할 핵심 규칙:
1. {knowledge_1.rule} — {knowledge_1.explanation}
2. {knowledge_2.rule} — {knowledge_2.explanation}
...
## 흔한 실수 (반드시 피할 것):
1. {pitfall_1.rule}
...
## 참고할 좋은 예시:
- {example_good_1}

[Layer 2 — 회사 맞춤 지식] (있으면)
## 이 회사의 과거 제안서 스타일:
...

[Layer 3 — 승패 분석] (있으면)
## 이 발주처의 평가 경향:
...

[RFP Context]
## 이번 공고 평가기준:
{rfp_evaluation_criteria}

[Task]
위 규칙과 컨텍스트를 반영하여 "{section_name}" 섹션을 작성하세요.
```

**Layer 1만으로도 작동하는 이유**: 보편 지식(구조, 작성법, 평가기준 이해, 시각화 규칙, 흔한 실수)만 있어도 "아무것도 모르는 AI"와는 차원이 다른 결과물이 나온다. Layer 2, 3은 정확도를 높이는 부스터이지, Layer 1이 기본 품질을 보장하는 핵심이다.

#### Step 4-A: Anti-Pattern Gate — 능동적 품질 차단 레이어

pitfall 카테고리를 단순 "참고 지식"이 아닌 **런타임 검증 레이어**로 승격:

```
[생성 단계] Layer 1 + 2 + 3 → 섹션 초안 생성

[검증 단계] Anti-Pattern Gate (pitfall DB 기반)
  ✗ "최고 수준의 기술력" → 근거 없는 추상 표현 감지 → 자동 재작성
  ✗ "당사 ○○시스템" → 블라인드 위반 감지 → 자동 마스킹
  ✗ 표 아래 설명 없음 → 자동 해설 문단 삽입
  ✗ 한 페이지에 메시지 3개 → "핵심 1개"로 분리 권고

→ 통과한 것만 최종 출력
```

이렇게 하면 "AI가 만든 제안서인데 감점 사유 0건"을 구조적으로 보장. §2-2-D 감점 회피 체크리스트와 결합하여 이중 방어.

#### Step 5: 품질 검증 — Golden Test Set

Layer 1 구축 후, 서비스 투입 전 검증:

1. 실제 나라장터 공고 10건 선정 (IT, 건설, 용역 등 다양하게)
2. 각 공고에 대해 Layer 1 지식 **있을 때 vs 없을 때** 제안서 생성
3. 비교 기준:
   - 평가기준 항목 커버리지 (빠뜨린 항목 수)
   - 구조 적절성 (전문가 기준)
   - 구체성 (숫자/방법론/일정 포함 여부)
   - 흔한 실수 발생 여부

**목표: Layer 1 적용 시 커버리지 30%+ 향상, 치명적 실수 0건**

#### Layer 1 실행 타임라인

| 주차 | 할 일 | 산출물 |
|------|--------|--------|
| 1주 | YouTube 50편 큐레이션 + 자막 추출 | raw_transcripts/ (50파일) |
| 1주 | 공식 문서 30편 수집 + 파싱 | raw_documents/ (30파일) |
| 2주 | LLM Pass 1: 지식 추출 | extracted_knowledge.jsonl (2,000+건) |
| 2주 | LLM Pass 2: 중복제거 + 보강 + confidence 계산 | knowledge_base.jsonl (800~1,500건) |
| 3주 | 벡터DB 2-Collection 구축 + 검색 테스트 | ChromaDB collection 2개 |
| 3주 | Golden Test 10건 평가 | evaluation_report.md |
| 4주 | 프롬프트 튜닝 + 품질 개선 | 프로덕션 투입 |

**비용 추정**: YouTube 자막(무료) + Whisper(자막 없는 영상 10편, ~$2) + LLM 지식 추출(GPT-4o-mini 100만 토큰, ~$5) + 임베딩(~$1) = **총 $10 미만으로 Layer 1 완성.**

**Quick Start — Mini Layer 1**: YouTube 20편 + 공식 가이드 10편만 먼저 → 2주 내 A-lite 데모와 동시 투입. 나머지 소스는 이후 점진적 추가 (파이프라인이 추가 소스 → 자동 정제 → 벡터DB 업데이트로 설계됨).

### 2-2. Layer 1에 녹여야 할 핵심 노하우 6가지

**A. 평가위원의 현실 이해 ("평가의 역설")**
- 평가위원은 제안서를 자세히 읽지 않는다 (시간이 없다)
- 보는 것: 그래픽, 캡션, 제목, 강조 문구, Executive Summary
- → AI 생성 시: 모든 페이지에 "이 페이지의 핵심 1줄" 캡션 자동 생성

**B. 배경-목적-비전의 주체 구분**
- 배경 = 최종고객 입장 ("시장이 이렇게 변하고 있어")
- 목적 = 발주사 실행부서 입장 ("이 사업으로 이걸 해결하고 싶어")
- 비전 = 제안사 입장 ("우리가 이렇게 만들어 드리겠습니다")
- → AI 생성 시: 각 섹션의 "화자"를 자동으로 구분하여 톤 조절

**C. 논리 구조 = "제안 나무(Recommendation Tree)"**
- 최상위: 핵심 주제 → 하위: 뒷받침 근거가 나뭇가지처럼 연결
- → AI 생성 시: 모든 주장에 "근거 → 수치 → 출처/진척도" 구조 강제

**D. 감점 회피 체크리스트**
- 블라인드 평가 위반 (회사명/로고 노출) → 자동 제거
- 설명 없는 표/차트 → 표마다 해설 문단 자동 생성
- 근거 없는 추상적 주장 → "최고 수준" 같은 표현 경고
- RFP 규격 위반 → 페이지 수, 폰트, 용지 사이즈 자동 검증
- 정책 방향 불일치 → 정부 정책 키워드 자동 삽입

**E. 정부지원사업 사업계획서 특화 (PSST 프레임워크)**
- Problem: 구체적 시장 데이터 + 실제 고객 인용
- Solution: 기술적 차별점 + 프로토타입/MVP 존재 여부
- Scale-up: TAM→SAM→SOM 시장 규모 + 구체적 시점별 계획
- Team: 핵심 인력의 "이 문제를 풀 수 있는" 직접 경험
- → AI 생성 시: 각 항목별 템플릿 + 자동 웹서치로 시장 데이터 삽입

**F. PPT 발표 전략**
- A3 가로형이 일반적 (공공기관)
- 2~4단 분할 레이아웃, 그래픽 50% 미만 유지
- 컬러 패널로 핵심 수치 강조
- "사례로 증명" — 유사 실적을 스토리텔링
- → AI 생성 시: 발표 시나리오(20분/30분) 자동 구성 + 슬라이드별 발표 노트

### 2-3. Layer 2 구축 — 회사별 맞춤 학습

사용자 온보딩 플로우:

**Step 1: 과거 제안서 업로드** (낙찰된 것 우선, 3~10건 권장)
- PDF/DOCX/HWP 자동 파싱

**Step 2: AI 자동 분석**
- 문체 분석: 어휘 수준, 문장 길이, 존칭 패턴
- 구조 분석: 섹션 구성, 목차 패턴, 페이지 배분
- 강점 표현: "우리 회사가 반복적으로 강조하는 것" 추출
- 시각화 스타일: 표/그래프 사용 패턴, CI 컬러
- 용어 사전: 회사 특유의 약어, 프로젝트명, 솔루션명

**Step 3: 맞춤 프로필 생성**
```
"귀사의 제안서 스타일을 분석했습니다:
  - 문체: 격식체, 평균 문장 28자
  - 강점: 인력 경험 강조, 기술 아키텍처 도식 활용
  - 구조: 사업이해(15%) → 기술접근(40%) → 관리(20%) → 실적(25%)
  - 특징: 비유/은유 활용, 정책 키워드 적극 반영"
```

**Step 4: 생성 시 적용**
- 기본 모델의 노하우 + 회사 맞춤 스타일 = 최종 출력

### 2-3-A. 수정 Diff 학습 루프 — "RLHF for Proposals"

AI가 제안서 초안을 생성하면 사용자가 반드시 수정한다. 이 **수정 내역 자체가 엄청난 학습 데이터**다.

```
AI 초안 생성 → 사용자가 수정 → diff 추출
    → "이 회사는 '최적화된 시스템'을 '검증된 시스템'으로 고쳤다"
    → "이 회사는 항상 Executive Summary를 직접 다시 쓴다"
    → Layer 2에 자동 반영
    → 다음 제안서에서는 처음부터 "검증된"으로 생성
```

**패턴 vs 일회성 구분 로직**:
```
1회 수정: 기록만 (학습 안 함)
2회 같은 패턴: 후보로 마킹
3회 이상: Layer 2에 자동 반영 + 사용자에게 알림
  "Kira가 학습했습니다: 앞으로 '최적화된' 대신 '검증된'을 사용합니다."
```

**수정률(Edit Rate) = 품질 KPI**:
```
제안서 생성 v1:  사용자 수정 45% → 품질 55%
제안서 생성 v5:  사용자 수정 20% → 품질 80%
제안서 생성 v10: 사용자 수정 8%  → 품질 92%
→ "Kira가 귀사를 학습한 정도: 92%"
```

이것이 **구독 해지 방어의 핵심**이다. "92%까지 학습한 AI를 버리고 다시 시작할 건가요?"

**Layer 2 학습 데이터 4가지 소스 종합**:
```
① 과거 제안서 업로드 (온보딩 시 일괄)
② 수정 diff 자동 수집 (매번 제안서 생성 시) ← 이것
③ 낙찰결과 자동 태그 (나라장터 API)
④ 네트워크 익명 패턴 (크로스-고객)
```

### 2-4. Layer 3 — 승패 분석 모델 (Win-Loss Model)

**핵심 아이디어**: "낙찰 vs 탈락" 태그를 붙여서 업로드하면, AI가 "이길 때와 질 때의 차이"를 학습한다.

사용자 경험:
```
"과거 제안서를 업로드해주세요. 낙찰/탈락 여부를 표시해주시면
 Kira가 귀사의 승패 패턴을 분석합니다."

→ 업로드: 낙찰 제안서 5건 + 탈락 제안서 3건
→ AI 분석:
    "낙찰 패턴 발견:
     - 사례 기반 증명이 3건 이상 포함됨
     - WBS가 주 단위로 상세 기술됨
     - Executive Summary에 수치가 5개 이상
     - 투입인력 경력이 해당 분야 10년+

    탈락 패턴 발견:
     - 유사 실적이 1건뿐
     - WBS가 월 단위로 뭉뚱그려짐
     - 추상적 표현 비율 40% 이상
     - 인력 경력이 범용적 (분야 특화 부족)"
```

**데이터 플라이휠 구조**:
```
사용자가 제안서 업로드 (낙찰/탈락 태그)
    → Layer 3 학습 데이터 증가
    → 승패 패턴 분석 정확도 향상
    → 더 좋은 제안서 생성
    → 낙찰률 증가
    → 사용자가 더 많은 제안서 업로드
    → 반복 (플라이휠)
```

**경쟁사 낙찰 제안서 학습**:
- 공공조달은 정보공개 청구로 낙찰 제안서 열람이 가능한 경우 있음
- Kira Bot이 안내: "이 공고의 낙찰 제안서를 정보공개 청구하시겠습니까?"
- 사용자가 획득한 경쟁사 제안서 업로드 → 익명화된 인사이트로 모든 고객에게 환원
- 결과: "이 발주처는 이런 스타일의 제안서를 선호합니다" 인사이트

### 2-5. 기술적 구현: RAG + 프롬프트 앙상블

```
[Layer 1 기본 모델 프롬프트]
  "당신은 공공조달 제안서 작성 전문가입니다.
   다음 규칙을 반드시 따르세요: {Layer 1 노하우 DB에서 검색된 규칙들}"

+ [Layer 2 회사 맞춤 프롬프트]
  "이 회사의 제안서 스타일: {Layer 2 분석 결과}
   과거 유사 제안서에서 이런 표현을 사용했습니다: {RAG 검색 결과}"

+ [Layer 3 승패 분석 프롬프트]
  "이 회사의 승패 패턴: {Layer 3 분석 결과}
   낙찰 제안서에서 공통적으로 나타나는 특징: {승패 패턴}
   이 발주처의 선호 패턴: {발주처별 분석 결과}"

+ [RFP 컨텍스트]
  "현재 RFP 분석 결과: {rfx_analyzer 출력}
   평가항목: {추출된 배점표}
   요구사항: {추출된 과업 범위}"

= 최종 제안서 섹션 생성
```

**왜 Fine-tuning이 아닌 RAG인가?**
- 새 제안서 추가 시 DB 업데이트만 하면 됨 (재학습 불필요)
- 비용 1/10 (Fine-tuning GPU 비용 vs RAG 벡터 DB)
- 출처 투명성 (어떤 과거 제안서에서 참조했는지 보여줄 수 있음)
- 나중에 품질 더 올리고 싶으면 Fine-tuning 추가 가능 (단계적 접근)

## §3. 8개 확장 모듈

### 입찰 라이프사이클 매핑

```
Phase 1: 발견         Phase 2: 분석        Phase 3: 준비                Phase 4: 제출
┌──────────────┐ ┌──────────────┐ ┌─────────────────────────┐ ┌──────────────┐
│ 공고 수집     │ │ RFP 분석      │ │ A. 기술제안서 자동 생성   │ │ G. 제출 체크  │
│ 관심 필터링   │ │ 자격요건 추출  │ │ B. 수행계획서/WBS 생성   │ │    리스트     │
│ 일괄 평가     │ │ GO/NO-GO     │ │ C. PPT 발표자료 생성     │ │              │
│              │ │ Q&A 채팅      │ │ D. 회사 역량 DB          │ │              │
└──────────────┘ └──────────────┘ │ E. 가격제안서 산출       │ └──────────────┘
    ✅ 완성          ✅ 완성       │ F. 실적/경력 기술서      │      🔮
                                  └─────────────────────────┘
                                  + H. 승률 대시보드
                                       🚀 핵심 확장 영역
```

### 모듈 A: 기술제안서 자동 생성기 (핵심 킬러 기능)

**문제**: 기술제안서는 50~200페이지. 평가항목에 맞춰 매번 처음부터 쓴다.

**워크플로우**:
```
RFP 분석 결과 (이미 있음)
    + 회사 역량 DB (모듈 D)
    + 제안서 템플릿 라이브러리
    + 3계층 학습 모델
    + 평가기준 매핑 엔진
    ──────────────────────
    = 제출 가능 수준의 제안서 (DOCX)
```

**구현 핵심**:
- RFP에서 추출한 평가항목별로 섹션 자동 구성
- 조달청 평가기준 매핑 (제안개요/업체역량/사업전략/사업관리/품질관리)
- 3계층 학습 모델로 "업계 베스트 프랙티스 + 우리 회사 색깔 + 승패 패턴" 융합
- 과거 제안서 라이브러리에서 유사 답변 재사용 (Loopio/DeepRFP 스타일)
- 출력: 평가항목에 맞춰 구조화된 Word(.docx) 문서

### 모듈 B: 수행계획서 / WBS 생성기

**워크플로우**:
```
RFP 과업내용 분석
    + 사업기간/규모 추출
    + 유사 프로젝트 패턴 DB
    ──────────────────────
    = WBS 자동 생성 + 간트차트 + 인력배치표
```

**구현 핵심**:
- RFP에서 과업범위, 산출물 목록, 사업기간 추출 (rfx_analyzer 확장)
- 표준 방법론(폭포수/애자일/반복점진적) 템플릿 적용
- WBS 분해 → 일정 자동 산출 → 간트차트 이미지 생성
- 투입인력 구성표 자동 배치 (PM/PL/개발/QA)
- 출력: 엑셀(WBS) + 이미지(간트차트) + Word 섹션

### 모듈 C: PPT 발표자료 자동 생성기

**워크플로우**:
```
생성된 제안서 초안 (모듈 A)
    + 평가항목별 핵심 메시지 추출
    + PPT 템플릿 (회사 CI 반영)
    ──────────────────────
    = 발표용 PPTX 자동 생성
```

**구현 핵심**:
- 제안서 각 섹션에서 "평가위원이 볼 핵심 3줄" 추출
- 슬라이드 구성: 표지 → 사업이해 → 추진전략 → 시스템구성도 → 일정 → 인력 → Q&A
- 배점 비중에 따라 슬라이드 수 자동 배분
- 다이어그램/아키텍처 도식 자동 생성 (Mermaid → 이미지)
- 회사 CI 컬러/로고 자동 적용, 발표 시나리오(20/30분) + 슬라이드별 발표 노트
- 예상질문 + 모범답변 자동 생성
- 출력: PPTX (python-pptx)

### 모듈 D: 회사 역량 DB (Company Knowledge Base)

**이것이 모든 모듈의 기반이 되는 이유**:
- 모듈 A (제안서): "유사 실적 3건을 실적기술서 양식으로"
- 모듈 B (수행계획서): "PM 후보 3명의 경력 자동 매핑"
- 모듈 C (PPT): "회사 CI + 대표 프로젝트 사진 삽입"
- GO/NO-GO (기존): 더 정확한 자격요건 매칭

**구성 요소**:
```
회사 기본정보 (사업자등록, 면허, 인증서, 자본금)
    + 수행실적 DB (프로젝트명, 기간, 금액, 고객사, 핵심기술)
    + 인력 Pool (이력서, 자격증, 경력사항)
    + 과거 제안서 라이브러리 (낙찰/탈락 태그 포함)
    + 회사 소개서 / 홍보자료
    ──────────────────────
    = 구조화된 회사 역량 벡터 DB
```

**구현 핵심**:
- 온보딩 위저드: 사업자등록증 OCR → 자동 입력
- 실적 업로드: 계약서/실적증명서 PDF → AI 구조화
- 인력 관리: 이력서 업로드 → 경력/자격증 자동 추출
- 벡터 검색: "클라우드 마이그레이션 관련 유사 실적" 시맨틱 검색

### 모듈 E: 가격제안서 / 원가산출 보조

- SW기술자 노임단가표 내장 (한국소프트웨어산업협회)
- RFP 추정가격 + 사업기간 + 투입인력 → 원가 자동 산출
- 적정 할인율 시뮬레이션 (추정가격 대비 투찰률)
- 출력: 엑셀 원가산출서 (.xlsx)

### 모듈 F: 실적기술서 / 경력기술서 자동 생성

- RFP에서 "별지 양식" 감지 → 해당 양식에 맞춰 자동 채움
- 유사실적을 "관련도 순" 정렬 (RFP 과업 vs 실적 유사도)
- 인력 배치 최적화: 평가 가점 극대화를 위한 인력 조합 추천
- 출력: DOCX

### 모듈 G: 제출 체크리스트 / 서류 검증기

- RFP에서 제출서류 목록 자동 추출
- 체크리스트 생성 → 사용자가 하나씩 체크
- 서류 업로드 시 파일명/형식 자동 검증
- 블라인드 평가 위반 자동 감지 (회사명/로고 노출)
- 마감일 D-day 알림

### 모듈 H: 입찰 이력 분석 / 승률 대시보드

- 입찰 참여 이력 DB (참여/낙찰/유찰)
- 승률 분석: 사업 유형별, 금액대별, 발주처별
- 경쟁사 낙찰 패턴 분석 (공공데이터 기반)
- 추천 엔진: "이번 달 승률 높은 공고 Top 10"
- 시각화 대시보드

## §5. 기술 아키텍처 확장

### 현재 → 확장 아키텍처

```
현재 파이프라인:
  document_parser → rfx_analyzer → matcher → chat_router

확장 파이프라인 (Agentic 멀티스텝):
  document_parser
      → rfx_analyzer (요건 추출 + 평가기준 매핑)
      → matcher (GO/NO-GO)
      → proposal_planner (제안 전략 수립 에이전트)
          → section_writer (섹션별 초안 작성 에이전트)
          → wbs_generator (WBS/일정 생성 에이전트)
          → personnel_matcher (인력 매칭 에이전트)
          → cost_calculator (원가 산출 에이전트)
      → document_assembler (DOCX/PPTX/XLSX 조립)
      → quality_checker (품질 검증 에이전트)
```

### Agentic 워크플로우 설계

Sequential Pipeline + Hierarchical Supervisor 패턴:

```
Supervisor Agent (proposal_planner)
    ├── Worker: section_writer × N (병렬, 섹션별)
    ├── Worker: wbs_generator (순차, 과업 분석 후)
    ├── Worker: personnel_matcher (순차, 인력 DB 조회)
    ├── Worker: cost_calculator (순차, 인력 배치 후)
    └── Assembler: document_assembler (최종 조립)

각 Worker는:
    1. 공유 메모리(RFP 분석 결과, 회사 DB)에 접근
    2. 자기 영역의 산출물 생성
    3. 결과를 공유 메모리에 저장
    4. Supervisor가 품질 검증 후 다음 단계로
```

### 기존 코드 활용 전략

| 기존 모듈 | 확장 활용 |
|-----------|----------|
| `rfx_analyzer.py` | 평가기준 매핑, 과업범위 추출 추가 |
| `matcher.py` ConstraintEvaluator | 회사 역량 DB 연동으로 정확도 향상 |
| `engine.py` ChromaDB + BM25 | 과거 제안서 라이브러리 검색 |
| `document_parser.py` | 회사 문서(실적증명서, 이력서) 파싱 |
| `proposal_generator.py` (rag_engine) | 확장하여 다중 섹션 생성기로 |
| `rfp_synonyms.py` | 제안서 작성 시 용어 통일에도 활용 |

## §6. 3단계 로드맵

### Phase 1: 기반 구축 (2개월) — "A-lite 먼저" 전략

**핵심 통찰**: 회사 역량 DB가 완성되기 전에도, **RFP 분석 결과 + Layer 1 기본 모델만으로 "범용 제안서 초안"은 생성 가능**하다. 회사 DB는 품질 부스터이지 필수 전제가 아니다. → **2주 만에 데모 가능**.

| 주차 | 모듈 | 내용 |
|:---:|------|------|
| Week 1-2 | **A-lite** (제안서 MVP) | Layer 1 기본 모델 + RFP 분석만으로 범용 제안서 초안 생성. 회사 DB 없이도 동작. "60점짜리 초안"이지만 **2주 만에 데모 가능** |
| Week 3-4 | **D** (회사 역량 DB) | 실적/인력/과거 제안서 구조화. 이것이 들어가면 60점 → 85점 |
| Week 5-6 | **A-full + G** | Layer 2 연동 + 회사 맞춤 제안서 + 제출 체크리스트. 85점 → 92점 |
| Week 7-8 | **수정 diff 학습 루프** | 사용자 수정 diff → 패턴 감지 → Layer 2 자동 업데이트. 쓸수록 좋아지는 구조 완성 |

**왜 이 순서인가?**
- D(회사 DB)부터 시작하면 → 2~3주 동안 눈에 보이는 결과물이 없다
- A-lite부터 시작하면 → 2주 만에 "RFP 넣으면 제안서 나온다" 데모 가능
- 회사 DB는 "초안 품질을 60점에서 92점으로 올리는 부스터"이지 "없으면 아무것도 안 되는 전제"가 아니다

Phase 1 완료 시:
```
"공고 검색" → GO 판정 → "제안서 초안 만들어줘"
    → Week 2: 범용 초안 (60점, Layer 1만)
    → Week 4: 회사 맞춤 초안 (85점, Layer 1+2)
    → Week 6: 고품질 제안서 + 체크리스트 (92점)
    → Week 8: 사용자 수정이 학습되는 자가 개선 루프
```

### Phase 2: 핵심 자동화 (2~3개월)

| 순서 | 모듈 | 이유 |
|:---:|------|------|
| 4 | B. 수행계획서/WBS | 제안서의 핵심 구성요소 |
| 5 | F. 실적/경력 기술서 | 반복 작업 자동화 효과 큼 |
| 6 | C. PPT 발표자료 | 발표 준비 시간 대폭 단축 |

Phase 2 완료 시 — 원클릭 입찰 패키지:
```
GO 판정 → 원클릭:
    📄 기술제안서 (DOCX, 100+ pages)
    📋 수행계획서 + WBS (XLSX + DOCX)
    🎯 발표자료 (PPTX, 25장)
    📋 실적기술서 / 경력기술서 (DOCX)
    ✅ 제출 체크리스트
```

### 학습 파이프라인 구축 로드맵 (Phase 1과 병행)

| 주차 | 작업 |
|:---:|------|
| Week 1-2 | 유튜브 50편 큐레이션 + 트랜스크립트 수집 → **A-lite 데모에 즉시 활용** |
| Week 2-3 | 블로그/공식 가이드 크롤링 + 구조화 |
| Week 3-4 | 노하우 DB 구축 (카테고리별 벡터 DB) + **회사 역량 DB와 동시 구축** |
| Week 4-5 | 기본 모델 프롬프트 엔지니어링 + 테스트 |
| Week 5-6 | 회사 맞춤 온보딩 파이프라인 구축 (Layer 2 연동) |
| Week 6-8 | 실제 RFP로 제안서 생성 테스트 + 품질 평가 + **수정 diff 학습 루프 구축** |

### 성공 지표

- 전문가 리뷰 시 "70% 이상 그대로 사용 가능" → 성공
- 생성 시간: RFP 업로드 후 10분 이내 50페이지 초안 완성
- 감점 요소 0건 (블라인드 위반, 규격 위반 등 자동 검증)

## §10-A. 인텔리전스 엔진 (브레인스토밍 확장)

### 나라장터 낙찰 결과 자동 수집 — Layer 3 자동화

나라장터 Open API에는 공고 수집 외에 **개찰 결과(`getOpenBidListInfoServc`)**와 **낙찰 결과(`getSucBidListInfoServc`)** API가 있다.

```
우리 고객이 공고 A에 입찰함
    → Kira Bot이 제안서 초안 생성
    → 2주 후, 나라장터에서 낙찰 결과 발표
    → Kira Bot이 자동으로 "공고 A: 낙찰" 또는 "공고 A: 탈락" 태그
    → Layer 3 자동 업데이트 (사용자 액션 제로)
```

추가 수집 가능 데이터:
- 낙찰 금액 → 적정 투찰가 예측 모델
- 참여 업체 수 → 경쟁률 예측
- 낙찰 업체명 → 경쟁사 분석

### 발주처 프로파일링 — IT 리터러시 난이도 조절

발주처마다 평가위원 풀의 성격이 다르다:

```
NIA 발주 → IT 리터러시 높음 → 기술 깊이 강조, 아키텍처 상세
지자체 발주 → IT 리터러시 낮음 → 비유/은유 많이, 그림으로 설명
국방부 발주 → 보안 민감도 높음 → 보안인증/물리적 보안 강조
교육청 발주 → 교육행정 전문가 → 교육 효과/사용자 편의 강조
```

구현: 발주처별 `audience_tech_level` 파라미터를 제안서 생성 프롬프트에 주입.
데이터 축적: 발주처별 낙찰 패턴 분석 → 프로파일 자동 업데이트.

### GO/NO-GO 2.0 — 승률 기반 의사결정

현재 GO/NO-GO는 "자격 충족 여부"의 이진 판단. 데이터가 쌓이면 확률 기반으로 업그레이드:

```
현재 GO/NO-GO:
  "자격 충족: YES → GO"

확장 GO/NO-GO:
  "자격 충족: YES
   예상 경쟁률: 8개사
   귀사 예상 기술점수: 64/70 (상위 30%)
   적정 투찰가: 2.64~2.76억 (추정가의 88~92%)
   종합 승률: 약 35%
   → GO (상위권 진입 가능, 기술 차별화에 집중 권장)"
```

기대수익 모델:
```
공고 A: 추정가 3억, 승률 35%, 예상 마진 15% → 기대수익 1,575만원
공고 B: 추정가 10억, 승률 15%, 예상 마진 10% → 기대수익 1,500만원
→ "공고 A가 금액은 작지만 기대수익이 더 높습니다"
```

### 입찰 포트폴리오 전략 — AI BD 디렉터

중소기업 BD팀은 2~3명. 월 최대 2~3건의 제안서만 작성 가능. "어디에 집중할 것인가"가 매출을 결정:

```
이번 달 귀사에 적합한 공고 23건 중:

🏆 Top Pick (승률 40%+, 강력 추천):
  ① NIA 클라우드 전환 사업 — 유사실적 5건, 기술 적합도 92%
  ② 교육부 LMS 구축 — 인력 PMP 2명 보유, 경쟁률 낮음

⚡ 도전 가치 (승률 20~40%):
  ③ 과기부 AI 분석 플랫폼 — 기술 적합 높지만 경쟁 치열

⚠️ 리스크 (승률 <20%, 신중 권장):
  ④ 국방부 보안시스템 — 보안인증 부족
```

### Post-Bid Learning Loop — 제출 후 자동 학습

```
제출 후 루프:
  ① 낙찰 결과 자동 수집 (나라장터 API)
  ② 낙찰/탈락 자동 태그 → Layer 3 업데이트
  ③ 탈락 시: "왜 떨어졌을까?" 자동 분석
      → 기술점수 추정 (발주처별 평균 대비)
      → 어떤 섹션이 약했을지 AI 추론
      → "다음에 이 유형 공고에서는 이걸 보강하세요" 피드백
  ④ 낙찰 시: 성공 패턴 Layer 3에 강화
  ⑤ 정보공개 청구 가이드
      → "낙찰 제안서를 열람해서 업로드하시면 비교 분석해드립니다"
```

입찰 성장 리포트 (분기별):
```
2026년 3분기 귀사 입찰 성과 리포트:
  ─ 참여 12건, 낙찰 4건 (승률 33%, 전분기 대비 +8%p)
  ─ 개선된 점: 기술점수 평균 62 → 65 (+3점)
  ─ 개선 권장: 인력 구성에서 PMP 자격 보유율 높이면 승률 +12%p 예상
  ─ 다음 분기 추천 공고 유형: SW 개발 (NIA 발주, 3~5억 규모)
```

### 네트워크 효과 — 입찰 시장의 데이터 플랫폼

```
사용자 증가 → 입찰 결과 데이터 축적 → 인사이트 정확도 향상
    → 새 사용자 유입 + 기존 사용자 이탈 방지 → 데이터 더 축적 → (반복)
```

익명화된 크로스-고객 인사이트:
- "이 유형의 공고에서 낙찰 기업들의 공통 특징: 유사실적 5건+, PMP 보유율 60%+"
- 제안서 원문은 절대 공유하지 않음 — 패턴만 익명화하여 환원

## §10-B. 민간 입찰 개방형 + 정부지원사업

### 공공 vs 민간 이원 구조

```
공공조달 (코어, 풀 자동화):
  나라장터 자동 수집 → 분석 → 제안서 → PPT → 체크리스트 → 낙찰결과 추적
  = 모든 파이프라인이 자동으로 돌아감

민간 입찰 (개방형, 수동 업로드):
  자동차/건설/방산 등 어떤 산업이든
  → "RFP 파일을 업로드하세요" → 분석 + 제안서 생성
  = 공고 수집/낙찰 추적은 없지만, 분석·생성 엔진은 동일
```

전략적 의미:
- 엔진은 하나, 공공은 자동화 프리미엄, 민간은 셀프서비스
- 민간 사용자가 만족하면 → "현대 SRM 포탈 자동 수집 기능 만들어주세요" 수요 발생
- 민간 RFP 데이터로 산업별 패턴 축적 ("민간은 가격 비중 50%", "자동차는 IATF 16949 필수")

## §10-C. Layer 1 부트스트래핑 전략

유튜브 노하우만으로는 부족하다. 3개 소스를 결합:

```
Layer 1 = 이론 (유튜브 50편 + 블로그 + 교육기관 커리큘럼)
        + 실전 성공 (조달청/NIA 우수 제안서 사례집)
        + 실전 실패 (탈락 사유 모음 — 입찰 커뮤니티 크롤링)
```

왜 실패 사례도 학습해야 하는가:
- 성공 사례만 학습하면 "왜 좋은지"는 알지만 "뭘 하면 안 되는지"는 모른다
- 탈락 사유 패턴: 블라인드 위반, 규격 위반, 실적 부족, 인력 부적합
- 감점 회피 기능의 정확도가 비약적으로 향상
