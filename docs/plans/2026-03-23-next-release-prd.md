# Kira Bot v2.0 PRD (Product Requirements Document)

**제품명:** Kira Bot -- 공공조달 입찰 AI 플랫폼
**버전:** v2.0 (Next Major Release)
**작성일:** 2026-03-23
**현재 상태:** Bid Studio Slice 1-5 완료, Production 배포(Railway), Staged Rollout 진행 중

---

## 1. Product Vision

Kira Bot은 대한민국 공공조달 시장에서 입찰 전 과정을 AI로 자동화하는 유일한 수직통합 플랫폼이다. 단순 문서 생성기를 넘어, "공고 발견 -> 적합성 판단 -> 회사 역량 매칭 -> 제출 패키지 생산 -> 검토/수정 -> 재학습"의 전체 수주 루프를 하나의 워크스페이스에서 완결하며, 사용할수록 해당 회사의 문체, 강점, 수주 패턴을 학습하여 경쟁사가 복제할 수 없는 회사별 AI를 구축한다. v2.0의 북극성은 **"첫 사용 시 제출 가능 수준의 입찰 문서를 만들고, 반복 사용 시 수주 확률을 높이는 플랫폼"**이다.

## 2. User Stories (MoSCoW 우선순위)

### Must Have (M)

**US-01. 적합 공고 자동 큐레이션**
> 입찰 담당자로서, 내 회사의 업종/지역/금액대/실적에 맞는 공고만 자동으로 추려서 보고 싶다.

수용 기준:
- 회사 프로필(업종, 지역, 금액대, 인증) 기반으로 나라장터 공고를 자동 필터링
- 매일 새 공고에 대해 적합도 점수(0-100)를 계산하여 상위 공고를 대시보드에 노출
- 알림 규칙에 없더라도 "이 공고는 회사에 맞을 수 있습니다" 추천 표시
- 적합도 계산에 회사 실적 이력(CompanyDB)과 과거 입찰 이력 반영

**US-02. 제출 가능 수준 품질 검증**
> 제안서 작성자로서, 생성된 문서가 RFP 요구사항을 얼마나 충족하는지, 누락된 항목이 무엇인지 한눈에 알고 싶다.

수용 기준:
- 생성 문서에 대해 RFP 요구사항 항목별 충족/누락 매트릭스 자동 생성
- 금지 표현(가격 암시, 부정 표현) 자동 탐지 및 경고
- 누락 항목에 대한 보강 추천(어떤 실적/인력/기술을 추가해야 하는지)
- 품질 리포트를 Review Stage에 통합 (별도 페이지 불필요)

**US-03. 팀 협업 워크플로우**
> 팀장으로서, 여러 담당자가 하나의 입찰 프로젝트에서 역할을 분담하고 리뷰/승인 흐름을 밟고 싶다.

수용 기준:
- 프로젝트별 팀원 초대 (owner/editor/reviewer/approver/viewer 역할)
- 문서별 코멘트 및 수정 요청 기능
- 승인 워크플로우 (editor -> reviewer -> approver 순서)
- 실시간 또는 근실시간 알림 (이메일/인앱)
- 활동 이력 타임라인 (audit log 기반)

**US-04. 다중 문서 일괄 생성**
> 제안서 작성자로서, 제안서/수행계획서/PPT/실적기술서를 개별이 아닌 패키지 단위로 한번에 생성하고 싶다.

수용 기준:
- Package Stage에서 "전체 생성" 버튼으로 자동화 가능 문서 일괄 생성
- 문서 간 일관성 유지 (동일 회사 컨텍스트, 동일 RFP 해석)
- 개별 문서 재생성 시 다른 문서에 영향 없음
- 일괄 생성 진행률 실시간 표시

**US-05. HWP/HWPX 출력 지원**
> 입찰 담당자로서, 공공기관에서 가장 많이 요구하는 HWP/HWPX 형식으로 문서를 직접 다운로드하고 싶다.

수용 기준:
- 제안서/수행계획서를 HWP 또는 HWPX 형식으로 다운로드 가능
- RFP에서 제공하는 HWP 템플릿에 생성 내용을 주입하는 방식 지원
- DOCX/HWP 선택 다운로드 (사용자 선택)
- 한글 폰트/레이아웃/표 서식 정상 렌더링

### Should Have (S)

**US-06. 승패 분석 및 학습 (Layer 3)**
> 대표로서, 과거 입찰 결과(낙찰/유찰)를 분석하여 어떤 유형의 공고에서 우리 회사가 강한지 파악하고 싶다.

수용 기준:
- 입찰 결과(낙찰/유찰) 입력 인터페이스
- 업종/금액대/계약방식별 수주율 통계 대시보드
- "왜 떨어졌는가" AI 분석 (제안서 대비 RFP 요구 매칭 역분석)
- 분석 결과를 향후 GO/NO-GO 판단에 자동 반영

**US-07. 회사 자산 버전 관리**
> 입찰 담당자로서, 실적/인력/인증 정보가 업데이트될 때 이전 버전과 비교하고, 어떤 프로젝트에서 어떤 버전을 사용했는지 추적하고 싶다.

수용 기준:
- CompanyDB 항목별 변경 이력 (생성/수정/삭제 시점 기록)
- 프로젝트별 "이 문서 생성에 사용된 회사 데이터 스냅샷" 열람
- 실적/인력 만료(유효기간) 경고
- 벌크 업데이트 (CSV/Excel 일괄 등록)

**US-08. 대시보드 -- 조직 KPI 뷰**
> 대표/팀장으로서, 우리 조직의 입찰 활동 현황(진행 중 프로젝트, 문서 생성 건수, 수주율)을 한 화면에서 보고 싶다.

수용 기준:
- 진행 중/완료/보관 프로젝트 수 카운트
- 이번 달 문서 생성 건수, 분석 건수
- 수주율 트렌드 (결과 입력 후)
- 팀원별 활동량 (프로젝트 참여, 문서 생성, 리뷰 건수)
- 다가오는 마감 공고 D-day 리스트

**US-09. 다국어 문서 생성 (영문 제안서)**
> 입찰 담당자로서, 외자 또는 국제 입찰에 필요한 영문 제안서/기술문서를 생성하고 싶다.

수용 기준:
- 생성 시 언어 선택 옵션 (한국어/영어)
- 영문 생성 시 공공조달 영문 용어 사전 적용
- 한-영 병행 생성 (한글 초안 -> 영문 번역 아님, 각각 독립 생성)

### Could Have (C)

**US-10. 전자입찰 연동 (나라장터 직접 제출)**
> 입찰 담당자로서, 완성된 문서를 나라장터에 직접 제출하는 과정을 간소화하고 싶다.

수용 기준:
- 나라장터 제출 규격(파일 형식, 파일명, 용량 제한) 자동 검증
- 제출 전 체크리스트 최종 확인 화면
- 나라장터 링크 원클릭 이동 (자동 제출은 API 미제공으로 불가)

**US-11. 오프라인 모드 (PDF 내보내기)**
> 현장 담당자로서, 인터넷 없이도 생성된 문서와 분석 결과를 열람하고 싶다.

수용 기준:
- 프로젝트 단위 전체 PDF 내보내기 (분석 결과 + 생성 문서 + 체크리스트)
- 내보낸 PDF에 메타 정보 (생성일, 버전, 사용된 회사 데이터) 포함

**US-12. 경쟁사 분석 (공개 정보 기반)**
> 입찰 담당자로서, 이 공고에 참여할 가능성이 높은 경쟁사의 공개 실적을 파악하고 싶다.

수용 기준:
- 나라장터 낙찰 이력 기반 경쟁사 목록 추출 (동일 발주기관/업종)
- 경쟁사 공개 실적 요약
- GO/NO-GO 판단 시 경쟁 강도 정보 제공

### Won't Have (이번 릴리스)

**US-13. 자동 가격 산정 엔진**
> 입찰 담당자로서, 투입 인력/기간/자재를 입력하면 적정 입찰가를 자동 산정하고 싶다.

사유: 가격 산정은 도메인 복잡도가 극히 높고 법적 리스크 존재. 향후 별도 모듈로 추진.

**US-14. 모바일 앱**
> 입찰 담당자로서, 이동 중에도 공고 알림을 확인하고 간단한 GO/NO-GO 판단을 하고 싶다.

사유: 웹 반응형 우선. 모바일 앱은 사용자 규모 1,000+ 이후 검토.

**US-15. 입찰 커뮤니티 / 마켓플레이스**
> 1인 입찰 컨설턴트로서, 다른 기업과 컨소시엄 파트너를 매칭하고 싶다.

사유: 플랫폼 네트워크 효과는 장기 비전. 현 단계에서는 단일 조직 완결성 우선.

| US-06 | 승패 분석 (Layer 3) | 5 | 5 | 0.6 | 6 | **2.5** | **P2** |

**RICE 계산식:** `(Reach * Impact * Confidence) / Effort`

**해석:**
- **P0 (즉시):** 품질 검증과 일괄 생성은 현재 Studio 파이프라인에 자연스럽게 붙으며, 사용자 전환율에 가장 큰 영향. 기존 코드 기반(quality_checker, package_classifier)이 있어 confidence 높음.
- **P1 (v2.0 포함):** 적합 공고 큐레이션은 Reach가 높으나 ML 파이프라인 필요. HWP 출력은 한국 공공조달 필수 요건. 팀 협업은 유료 전환의 핵심 동인.
- **P2 (v2.1):** 승패 분석은 Impact 최고이나 데이터 축적 필요. 회사 자산 버전 관리는 기존 DB 모델 확장.
- **P3 (v3.0):** 전자입찰 연동은 나라장터 API 제한. 경쟁사 분석은 데이터 소스 확보 어려움.

## 4. Technical Requirements (비기능 요구사항)

### 4.1 Performance

| 지표 | 현재 | 목표 | 비고 |
|------|------|------|------|
| RFP 분석 응답 시간 | ~15초 | < 10초 (p95) | 스트리밍 중간 결과 표시 |
| 단일 문서 생성 | ~30초 | < 20초 (p95) | 섹션 병렬 생성 |
| 일괄 생성 (4문서) | N/A | < 90초 (p95) | 문서별 병렬 처리 |
| 페이지 초기 로드 | ~3초 | < 1.5초 | Vite chunk 분할 (현재 index.js 1,235KB) |
| 나라장터 검색 | ~2초 | < 2초 유지 | API 응답 시간 외부 의존 |
| DB 쿼리 | < 100ms | < 50ms (p95) | 인덱스 최적화, 커넥션 풀 |

### 4.2 Security

| 요구사항 | 현재 상태 | 목표 |
|----------|----------|------|
| 인증 | 세션 기반 (web_saas NextAuth) | OAuth 2.0 / OIDC 표준 (Google, Kakao) |
| 인가 | require_project_access (RBAC) | 유지 + 팀 기반 ACL 확장 |
| 데이터 격리 | org_id 기반 멀티테넌시 | RLS(Row Level Security) 추가 적용 |
| 파일 업로드 보안 | Magic bytes + path traversal guard | 유지 + 바이러스 스캔 (ClamAV) |
| API Rate Limiting | 분석 5회/분, 업로드 5회/분 | 유지 + plan tier별 차등 |
| 감사 로그 | AuditLog 테이블 기록 | 유지 + 로그 보존 기간 정책 (90일) |
| 데이터 암호화 | TLS in transit | at-rest 암호화 (DB, 파일 스토리지) |
| KISA 보안 인증 | 미적용 | ISMS-P 요구사항 대비 점검 |

### 4.3 Scalability

| 측면 | 현재 아키텍처 | v2.0 목표 |
|------|-------------|----------|
| 동시 사용자 | 단일 인스턴스 (Railway) | 100 동시 사용자 |
| DB | 단일 PostgreSQL | Read replica 1개 + Connection pooling (PgBouncer) |
| 파일 저장 | 로컬 디스크 (data/) | S3-compatible 객체 스토리지 (R2/S3) |
| RAG Engine | 단일 ChromaDB 인스턴스 | 회사별 ChromaDB 파티션 (이미 구현) |
| LLM 호출 | 단일 OpenAI API 키 | 멀티 프로바이더 (OpenAI + Anthropic fallback) |
| 큐잉 | 동기 처리 | 비동기 작업 큐 (Celery/ARQ) -- 장시간 생성 |
| 모니터링 | 서버 로그 | Structured logging + Sentry + Datadog/Grafana |

### 4.4 Reliability

- **가용성:** 99.5% uptime (월간 ~3.6시간 다운타임 허용)
- **백업:** 일 1회 자동 DB 백업 + 7일 보존
- **장애 복구:** RTO < 1시간, RPO < 1시간
- **자동 복구:** Health check + 자동 재시작 (Railway/Docker)
- **그레이스풀 디그레이드:** LLM API 장애 시 캐시된 분석 결과로 부분 동작

### 4.5 Compliance

- 개인정보보호법 준수 (회사 데이터 내 개인정보 마스킹)
- 나라장터 API 이용약관 준수 (호출 빈도 제한, 데이터 캐싱 정책)
- 생성 문서 내 AI 생성 표기 정책 (투명성)

| KPI | 정의 | 목표 | 측정 방법 |
|-----|------|------|----------|
| **분류 정확도** | 패키지 분류기 정답률 | > 90% | manual override 비율의 역수 |
| **생성 수용률** | 사용자가 생성 결과를 수정 없이/소량 수정으로 사용하는 비율 | > 50% | edit_distance < 0.3인 비율 |
| **재학습 사용률** | relearn stage 진입 프로젝트 비율 | > 20% | audit_log WHERE action='style_skill_relearned' |
| **품질 경고 해결률** | 품질 검증 경고 -> 사용자 조치 비율 | > 70% | 경고 후 문서 재생성/수정 비율 |
| **TTFV (Time to First Value)** | 가입 -> 첫 문서 다운로드까지 시간 | < 15분 | 가입 시점 -> 첫 asset download 시점 |

## 7. Competitive Moat (방어 가능한 경쟁 우위)

### 7.1 한국 공공조달 시장 컨텍스트

한국 공공조달 시장 규모는 연간 약 180조원(2025년 기준). 입찰에 참여하는 중소기업은 약 12만개사이며, 연간 평균 5-20건의 입찰에 참여한다. 각 입찰당 제안서 작성에 평균 2-4주, 비용 500만-3,000만원이 소요된다.

현재 경쟁 환경:
- **나라장터/KONEPS:** 공고 조회 + 전자입찰 인프라. 문서 생성 기능 없음.
- **비드프로/비드마스터:** 공고 모니터링 + 단순 알림. AI 분석/생성 없음.
- **범용 AI (ChatGPT, Claude):** 범용 문서 작성 가능하나 공공조달 도메인 전문성 없음. RFP 분석, 패키지 분류, 회사 역량 매칭 불가.
- **SI 업체 자체 도구:** 사내용, 범용화 불가.

### 7.2 Kira Bot의 경쟁 해자

**해자 1: 공공조달 도메인 특화 AI 파이프라인**

범용 AI가 대체할 수 없는 것:
- RFP 다중패스 분석기 (RFxAnalyzer): 자격요건, 평가기준, 납품조건, GO/NO-GO를 구조화
- 패키지 분류기: (domain x method) 조합별 제출 패키지 자동 식별 (18개 케이스 코퍼스)
- 나라장터 API 네이티브 연동: 실시간 공고 조회 + 첨부파일 자동 파싱 (HWP/HWPX/PDF)
- 공공조달 전문 용어 사전 및 표현 패턴: Layer 1 (495 유닛 지식 베이스)

이것을 복제하려면 공공조달 도메인 전문가 + AI 엔지니어링 + 나라장터 데이터 축적이 동시에 필요하며, 최소 6-12개월의 진입 장벽.

**해자 2: 회사별 누적 학습 (Switching Cost)**

- Layer 2 (auto_learner): 사용자 편집 패턴에서 회사별 선호 문체/표현/구조를 자동 학습
- CompanyDB: 실적, 인력, 인증, 기술 역량이 축적될수록 문서 품질이 향상
- 프로젝트 히스토리: 과거 입찰 문서 + 분석 결과 + 재학습 데이터가 쌓일수록 다른 도구로 이전 불가
- Style Skill: 회사별 문서 스타일이 버전 관리되며, 사용할수록 정교해짐

전환 비용 추정: 6개월 사용 시 평균 50+ 프로젝트, 200+ 문서, 500+ 편집 패턴 축적. 경쟁 제품으로의 이전 시 이 자산을 재구축하는 데 3-6개월 소요.

**해자 3: 수직통합 워크플로우**

경쟁사들은 파이프라인의 일부만 커버:
```
[공고 발견] -- 비드프로/비드마스터가 여기만 커버
[RFP 분석] -- 없음
[적합성 판단] -- 없음
[회사 매칭] -- 없음
[문서 생성] -- 범용 AI가 여기만 커버 (품질 낮음)
[품질 검증] -- 없음
[재학습] -- 없음

Kira Bot = 전 구간 커버
```

단일 플랫폼에서 전 과정을 처리하므로 데이터가 끊기지 않고, 앞 단계의 결과가 다음 단계의 품질을 높이는 복합 효과 발생.

**해자 4: 네트워크 효과 잠재력 (Phase D 이후)**

- 많은 회사가 사용할수록 업종/지역/금액대별 수주율 벤치마크 데이터 축적
- 벤치마크 데이터는 새 사용자의 GO/NO-GO 판단 정확도를 높임
- 데이터 비대칭: 먼저 시장을 점유한 쪽이 데이터 우위를 가져감
