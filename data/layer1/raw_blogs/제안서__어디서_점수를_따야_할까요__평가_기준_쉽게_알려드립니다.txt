# Title: 제안서, 어디서 점수를 따야 할까요? 평가 기준 쉽게 알려드립니다
# Category: evaluation
# Quality: 상

제안서, 어디서 점수를 따야 할까요? SW 개발 입찰 평가 기준 쉽게 알려드립니다.
"제안서는 써야겠는데, 어디에 힘을 줘야 할지 모르겠어요..."
입찰 제안서를 처음 접한 분들이 자주 하는 말입니다.
이럴 땐 가장 먼저 "어떤 기준으로 점수를 주는지"알아야 해요.
2024년부터 공공 SW 개발사업에서 사용하는 평가 기준은 이전보다 더 구체적이고 자세해졌습니다.
이 글에서 쉽게 풀어드릴게요!
평가 기준은 총 6가지 영역으로 나뉘어요
제안서에서 평가자가 보는 기준은 아래 6가지예요.
| 평가영역 | 배점(100점 중) |
| 1. 전략과 방법론 | 15점 |
| 2. 기술 및 기능 | 24점 |
| 3. 성능과 품질 | 21점 |
| 4. 프로젝트 관리 | 15점 |
| 5. 프로젝트 지원 | 15점 |
| 6. 협력과 계약 | 10점 |
☞ 총 100점!
☞ 점수가 높은 항목일수록 더 정성껏 써야겠죠?
항목별로 쉽게 풀어 볼게요
1. 전략과 방법론(15점)
- 제안하는 시스템을 잘 이해하고 있나요?
- 어떤 전략으로 사업을 추진하나요?
- 어떤 기술을 쓸 계획인가요?
☞ 단순히 기술을 나열하지 말고, 왜 이 기술을 선택했는지, 어떤 문제를 해결하려는지 써주세요.
2. 기술 및 기능(24점)
- 시스템이 어떤 기능을 해야 하나요?
- 데이터, 보안, 운영 조건까지 다 포함되었나요?
☞ 이 항목이 가장 많은 점수를 차지해요!
→ "어떻게 만들 건지"뿐 아니라
→ "운영할 때 어떤 문제가 생길 수 있고, 어떻게 대비할 건지"까지 써야 합니다.
3. 성능과 품질(21점)
- 속도는 빠를까요?
- 다른 시스템과 잘 연결되나요?
- 문제없이 작동할 수 있을까요?
☞ 이 항목은 "좋은 품질의 시스템"인지 보여주는 부분입니다.
→ 테스트 계획, 장애 대응도 함께 제시하면 좋습니다.
4. 프로젝트 관리(15점)
- 일정은 어떻게 짤 건가요?
- 어떤 방식으로 프로젝트를 관리하나요?
- 필요한 장비는 확보되었나요?
☞ 평가자는 "말로만 잘하겠다"보다 "계획이 있고 준비된 기업인가?"를 봅니다.
→ Gantt 차트, 조직도 있으면 더 좋아요.
5. 프로젝트 지원(15점)
- 교육은 누가 어떻게 해주나요?
- 유지보수는 어디까지 지원하나요?
- 보안이나 장애에 대비한 방안은 있나요?
☞ 이건 제안서 중 운영 이후까지 신경 썼는지 보는 항목이에요.
6. 협력과 계약(10점)
- 중소기업과 어떻게 협력하나요?
- 하도급 구조는 투명한가요?
☞ 단순히 "협력하겠습니다."가 아니라 "누구와 어떻게 협력할지" 구체적으로 써주세요.
실무에서 이렇게 대응해 보세요!
| 항목 | 제안서에서 이렇게 써보세요 |
| 전략 및 방법론 | 고객의 문제를 제대로 이해했다는 걸 보여주세요. |
| 기술 및 기능 | 보안/운영까지 고려한 기능 설명이 좋습니다. |
| 성능 및 품질 | 테스트 계획과 품질 보증 방안이 핵심! |
| 프로젝트 관리 | 일정표, 투입 인원, 리스크 대응 방안 제시 |
| 프로젝트 지원 | 유지관리 계획, 교육 방법까지 포함 |
| 협력 및 계약 | 협력사 역할, 하도급 계획 구체화 |
한눈에 요약!
| 평가 항목 | 포인트 |
| 전략 및 방법론 | 왜 이 기술을 쓰는지 설명 |
| 기술 및 기능 | 만들고 운영할 수 있는 실력 보여주기 |
| 성능 및 품질 | 안정성과 연결성 강조 |
| 프로젝트 관리 | 실행 가능한 일정과 팀 구성 |
| 프로젝트 지원 | 운영 이후까지 고려한 계획 |
| 협력 및 계약 | 실질적 협력과 공정성 |
평가 기준을 알면 제안서가 달라집니다.!
이제 "제안서 잘 쓰는 법"이 아니라, "평가자가 보기 좋은 제안서"를 써야 할 때입니다.
