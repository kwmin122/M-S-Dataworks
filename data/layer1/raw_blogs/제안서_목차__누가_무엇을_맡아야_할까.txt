# Title: 제안서 목차, 누가 무엇을 맡아야 할까?
# Category: structure
# Quality: 상

제안서 작업, 제대로 하려면 역할 분담부터 시작하세요!
제안서를 만들 때 제일 먼저 하는 일 중 하나가 바로 '목차'를 정하는 일입니다.
목차는 제안서 전체의 뼈대이기 때문에, 잘 짜여져야 내용도 알차게 구성됩니다.
그런데 많은 팀이 이 목차를 '누가 어떤 부분을 맡아서 작성할지' 미리 정하지 않아,
막상 작성할 때 혼선이 생기는 경우가 많습니다.
그래서 오늘은 제안서 목차를 제대로 만들기 위해 팀에서 어떤 사람에게 어떤 역할을 맡겨야 하는지를 쉽게 설명해 드릴게요.
제안서에서 중요한 두 역할
제안서 작성할 때 보통 크게 두 부류의 사람이 함께 작업합니다.
1. 제안 책임자(PM 또는 PL)
전체 제안서를 총괄하는 사람입니다.
| 해야 할 일 | 설명 |
| 전체 흐름 파악 | 목차가 자연스럽게 연결되도록 구성하기 |
| 담당자 지정 | 각 항목별로 누가 작성할지 정하기 |
| 핵심 키워드 정리 | RFP(제안요청서)를 읽고 중요한 단어 추출 |
| 점검표 만들기 | 누락된 내용은 없는지 체크리스트로 확인 |
| 일정 관리 | 제안서 제출 일정을 고려하여 일정 수립 |
2. 제안서 팀원
각 홍목을 실제로 작성하는 실무자입니다.
| 해야 할 일 | 설명 |
| RFP 읽고 이해하기 | 어떤 내용을 요구하는지 파악하기 |
| 세부 목차 만들기 | 항목을 더 구체적으로 나누기(ex : 기능별로) |
| 내용 작성 | 요구사항에 맞게 솔루션, 효과 등을 작성하기 |
| 체크리스트 만들기 | 스스로 작성한 내용 점검용 |
예를 들어 설명해 볼게요
만약 "디지털트윈 시스템 기능 구성"이라는 항목이 있다면,
- PM은 이 항목이 전체 목차에서 어떤 위치에 있어야 할지 정하고, 누가 이걸 작성하면 좋을지 정합니다.
- 팀원은 "디지털트윈 시스템"에 대해 어떤 기능이 필요한지 정리하고, 실제로 내용을 작성합니다.
잘 작성하려면 이런 팁도 있어요
- RFP에 있는 요구사항은 빠짐없이 넣기
→ 평가 기준과 직접 연결되기 때문이에요. - 목차는 4단계 이하로 나누는 게 좋아요
→ 너무 깊게 나누면 읽기 어려워집니다. - 목차만 봐도 어떤 내용인지 알 수 있게 작성해요
→ex : "1. 시스템 개요" 보다 "1. 디지털트윈 시스템 구성 및 기대효과"가 더 좋죠.
실제 사례
어떤 공공기관 제안서에서는 다음과 같이 역할을 나눴어요.
- PM : 전체 목차 구성, 담당자 지정, 체크리스트 제작
- 개발팀장 : 시스템 구성, 기능 설명 파트 작성
- 기획자 : 제안서 내 들어갈 화면 설계 작성
- 디지이너 : 기획자 화면설계 바탕 예시 이미지 작성
- 제안팀 : 보안, 교육계획, 유지관 등 작성
이렇게 역할을 나누고, 작성 전부터 체크리스트를 만들어 시간도 절약하고, 빠진 내용도 없었어요.
마무리 정리
제안서를 작성할 때 목차 작성은 시작이자 핵심입니다.
이걸 제대로 하려면 단순히 항목을 나열하는 게 아니라
'누가 무엇을 언제까지 작성하는지' 명확히 정하는 것부터 시작해야 합니다.
목차 작성이 잘 되어 있으면 제안서 전체의 흐름도 깔끔해지고, 평가 점수도 좋아집니다.
