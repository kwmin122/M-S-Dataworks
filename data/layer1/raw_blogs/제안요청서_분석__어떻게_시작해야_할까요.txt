# Title: 제안요청서 분석, 어떻게 시작해야 할까요?
# Category: strategy
# Quality: 상

입찰 초보도 할 수 있는 4단계 분석법
"제안요청서(RFP)를 받았는데 뭐부터 보면 좋을까요?"
입찰 제안서를 처음 써보려는 분이라면, 이 질문을 한 번쯤 해보셨을 거예요.
RFP는 그 자체가 방대한 문서라 읽는 것도 버겁고, 막상 읽어도 '무엇을 써야 하는지' 알기 어렵죠.
하지만 괜찮습니다.
딱 4단계로 나누면 생각보다 쉬워요!
RFP 분석, 이렇게 하면 됩니다.
제안요청서를 분석하는 방법은 아래 4단계입니다.
1. 누가 만들었는지 보기(담당자 파악)
2. 문서 전체 구조 파악하기(구성체계)
3. 요구사항 뽑아내기
4. 우리만의 해결책 만들기
이제 각 단계의 예시와 함께 하나씩 풀어볼게요!
1. 누가 만들었는지 보기(담당자 역할 파악)
이 문서를 만든 사람(발주기관)은 어떤 입장일까요?
- 실무자? → 기능 중심 요구가 많아요.
- 관리자? → 전략, 예산, 성과에 관심 많아요.
예를 들면,
정보화팀에서 작성한 제안요청서는 "어떤 시스템이 필요하다", "어떤 화면이 있어야 한다"처럼
기능 요구가 자세히 쓰여 있을 가능성이 높아요.
문서를 만든 사람의 '의도'를 먼저 파악해 보세요.
2. 문서 전체 구조 파악하기(구성체계 이해)
제안요청서는 보통 다음과 같이 구성돼요.
∙ 사업 개요
∙ 현황 및 문제점
∙ 추진 계획
∙ 기술 요구사항
∙ 평가 기준
전체 목차를 한눈에 보고
"어디가 중요하고, 어디서 점수를 딸 수 있을지"미리 체크해 보세요.
예시)
기술 요구사항 파트가 길고 자세하다 → 여기서 많은 점수를 줄 가능성!
3. 요구사항 뽑아내기(요구사항 도출)
이제 문서를 읽으며 발주자가 진짜로 원하는 것을 찾아야 합니다.
∙ "반드시"라는 단어가 있으면 필수예요
∙ "선택사항", "추가 기능"은 옵션이에요
∙ 기능, 보안, 화면, 교육... 빠짐없이 체크
Tip : 요구사항을 표로 정리해 보세요.
| 요구 항목 | 필수/선택 | 위치 | 대응 여부 |
| 모바일 지원 | 필수 | 기능요구사항 3-2 | 가능 |
4. 우리만의 해결책 만들기(해결방안 정리)
이제 마지막!
도출한 요구사항을 기반으로 우리가 어떻게 대응할 것인지 정리합니다.
∙ 어떤 기술을 쓸 건지
∙ 비슷한 프로젝트 경험이 있는지
∙ 예산, 일정, 인력 계획은 어떻게 구성한 건지
예시)
"모바일 대응" 요구에 대해,
→ "기존 프로젝트에서 Flutter 기반 모바일 대응 경험 있음, 주요 화면은 ○○방식으로 구성 계획"
☞ 이게 바로 제안서의 뼈대가 됩니다.
한눈에 보는 분석 요약
| 단계 | 핵심 질문 |
| 1 단계 | 누가 이 문서를 썼을까?(담당자 파악) |
| 2 단계 | 문서 구조는 어떻게 되어 있지? |
| 3 단계 | 뭘 원하고 있지?(요구사항 도출) |
| 4 단계 | 우리는 어떻게 해결할까?(해결방안) |
이걸 그대로 팀원들과 공유하면,
기획자, 디자이너, 개발자와의 협업도 쉬워집니다.
마무리 한마디
제안서 작성은 "글쓰기"가 아니라
"이해 → 분석 → 제안"의 흐름이에요.
처음 분석을 잘해두면 제안서가 훨씬 쉽게, 빠르게 써집니다.
